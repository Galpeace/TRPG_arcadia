# シナリオ運用とシナリオ仕様書

## このファイルの位置

シナリオ仕様書の単位・構造、章類型カタログ、章開始プロセス、章末プロセス、周回最終章システム、不達トリガーを定める。OPUSのシナリオ設計ロール（章開始時・章末時）の運用基盤となる。

## 三秒で読む要約

章は単位で設計される。仕様書は単一の.md。章類型は時間の質で六分類。章末は物語的エンドロール・対外的評価・システム処理・出力選択・次章方向性の五段階。

## 一分で読む要約

このTRPGでは、シナリオはシリーズ単位で事前設計せず、章単位で設計される。OPUSは章開始前にPCのメインクエスト群とPCの現状を確認し、章類型（緊迫/探索/対峙/日常/内省/移動）を判定し、シナリオ仕様書を単一の.mdとして作成する。章開始時、プレイヤーには舞台・環境・あらすじが開示される。仕様書全体も同時に出力されるが、プレイヤーは違和感を覚えた時にのみ参照することが推奨される。章末は五段階で処理され、評価描写では評価の非対称を積極的に取り込む。プレイヤー宣言またはOPUS提案により次の章を周回最終章と決定するシステムを持つ。PCがシナリオ中核タスクを拒否した場合、別ルートで誘導せず章を終了する。

# 1. シナリオ仕様書の単位

## 1.1 章単位の設計

```yaml
原則:
  シナリオはシリーズ単位で事前設計せず、章単位で設計する。
  
  含意:
    - 次の章だけを設計する
    - 章末で次章を考える
    - PCの実際の選択を反映できる
    - シリーズ全体の縦糸は事前に固定しない
```

## 1.2 設計の起点

```yaml
設計の起点:
  シリーズ縦糸ではなく、以下を起点とする:
    - PCが持つメインクエスト群と各段階
    - PCの現状（状況、関係性、評価、装備）
    - 前章の結末と次章方向性選択肢でのプレイヤーの選択
  
  根拠:
    - 中核テーマ「世界は固定的、PCは不確定な因子」
    - PCのメインクエストが章の方向性を決める主因子
    - シリーズ縦糸を固定すると、PCの選択が縦糸に縛られる
```

## 1.3 「シリーズ」概念の扱い

```yaml
シリーズ概念:
  プレイ単位としてのシリーズ:
    - 一つのPCのプレイ全体を緩やかに「シリーズ」と呼ぶ
    - 明示的な話数の事前計画はしない
    - 周回最終章システム（後述）で終わり方を決める
  
  廃止された旧概念:
    - シリーズ縦糸（事前設計されるテーマ・伏線・黒幕）
    - 想定話数（3〜5話の事前計画）
    - 各話のログライン事前列挙
  
  代替:
    - PCのメインクエスト群が事実上の「縦糸」として機能する
    - 黒幕や伏線は、ワールド・クエストの自律進行や、
      PCの行動から浮上する形で立ち上がる
```

# 2. 章類型カタログ

章は「PCがどう過ごす時間か」で分類する。クエスト体系とは独立した軸である。

## 2.1 六つの類型

```yaml
章類型（時間の質ベース）:

  緊迫:
    定義: 時間的圧力・身体的危険が章を支配する
    含む状況: 戦闘、追跡、緊急救出、災害対応、脱獄、逃亡
    特徴:
      - 章の中心は明確
      - 不達トリガーが重要
      - PCの即時の判断が物語を動かす
  
  探索:
    定義: 未知への接近、情報・場所・人の発見が章を支配する
    含む状況: ダンジョン、聞き込み、調査、追跡（情報的）
    特徴:
      - 手掛かりの配置が重要
      - クエスト浮上の機会が多い
      - PCの主体的な調査が章を駆動する
  
  対峙:
    定義: 他者との社交・交渉・対立が章を支配する
    含む状況: 交渉、宴会、会議、対決、信頼構築、敵対の表明
    特徴:
      - 主要NPCの本心が重要
      - 関係性の変化が章の主軸
      - 評価変動が起きやすい
  
  日常:
    定義: 安全圏での過ごし、休息、雑事が章を支配する
    含む状況: 休息、買い物、馴染みの場所、街の散策、食事
    特徴:
      - 非クエスト要素が章を構成する
      - サブクエストが章の中心になることがある
      - クエスト浮上の素地が多い
  
  内省:
    定義: PCの内面・過去・関係性が章を支配する
    含む状況: 思索、過去との対面、価値観の問い直し、関係性の整理
    特徴:
      - PCの動機の変質が起きうる
      - 周囲の事件は背景に退く
      - 章末の評価変動は少ない
  
  移動:
    定義: 場所から場所への移動が章を支配する
    含む状況: 長旅、道中、辺境への往路復路
    特徴:
      - 環境変化が物語を作る
      - 道中の出会いと別離
      - 内省機会を伴うことが多い
```

## 2.2 類型の判定手順

```yaml
類型判定の手順（OPUSが章開始前に実行）:

  ステップ1: PCの状況を確認する
    確認項目:
      - PCに時間的圧力があるか
      - PCに身体的圧力があるか
      - PCに関係的圧力があるか
      - PCに状況的圧力があるか
      - PCが安全圏にいるか
      - PCに移動の必要があるか
  
  ステップ2: PCのメインクエストの段階を確認する
    確認項目:
      - 決算段階に近いメインクエストがあるか
      - 介入段階にあるメインクエストがあるか
      - 認識・理解段階のメインクエストのみか
      - メインクエストが保留状態か
  
  ステップ3: 前章の結末と次章方向性選択を確認する
    確認項目:
      - プレイヤーが選んだ次章方向性
      - 前章の自然な続きとして何が想定されるか
  
  ステップ4: 類型を決定する
    優先順位:
      1: 圧力があれば緊迫型を検討
      2: 主体的な調査・探索の意思があれば探索型を検討
      3: 重要NPCとの対峙が予定されていれば対峙型を検討
      4: PCが安全圏で休息を選んだら日常型を検討
      5: PCの内面に動きがあれば内省型を検討
      6: 場所移動が章の主体なら移動型を検討
    
    混合可能性:
      - 単一類型に収まらない場合は主類型を決め、副要素を任意層に書く
      - 例: 「主類型: 探索、副要素: 対峙の機会あり」
```

# 3. 章の中心

章の中心は、章類型とは別の軸である。

## 3.1 章の中心となりうる要素

```yaml
章の中心:
  
  メインクエスト:
    - PCにとって重要なクエスト（2R-quest.md参照）
    - 緊迫型・探索型・対峙型の章では中心になりやすい
  
  サブクエスト:
    - PCにとって些細なクエスト
    - 日常型の章で中心になることがある
    - 例: 友人の酒屋のネズミ退治だけで終わる日常回
  
  非クエスト要素:
    - 関係深化、文化接触、内省、副次的調査、日常体験
    - 内省型・移動型・日常型の章で中心になることがある
    - 例: PCが過去と向き合うだけの内省章
```

## 3.2 メイン/サブと章の中心の独立性

```yaml
独立性の原則:
  メイン/サブはPCにとっての重要度の軸である。
  章の中心はその章の主役の軸である。
  両者は独立して動く。

例:
  - メインクエスト「弟を探す」が章の中心 → 通常の主軸進展回
  - サブクエスト「酒屋のネズミ退治」が章の中心 → 日常回
  - 非クエスト要素「過去との対面」が章の中心 → 内省回
  - メインクエスト「弟を探す」が章中に動かず、非クエスト要素が中心 → 休息回
```

## 3.3 中心の不在

```yaml
中心の不在:
  章に明確な中心がない場合もある:
    - 純粋な移動の章
    - 何も起きない日常の章
    - 複数の要素が並列する章
  
  この場合の処理:
    - 主類型に応じて章を構成する
    - 章の中心欄に「中心なし」と記載する
    - これは正規の状態である
```

# 4. シナリオ仕様書の構造

## 4.1 基本仕様

```yaml
基本仕様:
  形式: 単一の.mdファイル
  サイズ: 200〜500行（密度中）
  フォーマット: yaml構造化＋散文の混合
  
  yaml構造化を使う箇所:
    - メタ情報
    - 章類型
    - PCの現状
    - 章の中心
    - 主要NPC（公言/本心）
    - 終わり方の構造
  
  散文で書く箇所:
    - エピグラフ
    - 舞台描写
    - あらすじ
    - 物語的な含みのある記述
```

## 4.2 必須層

```yaml
必須層（すべての章で記述）:

  メタ情報:
    - 章タイトル
    - エピグラフ（短い引用句）
    - 章類型（緊迫/探索/対峙/日常/内省/移動）
    - 想定シーン数
  
  舞台:
    - 場所、地域、文化的特徴
    - 環境条件（天候、時間帯、季節、光量）
  
  あらすじ:
    - 章の入り口の状況
    - 初期配置
  
  PCの現状:
    - 現在のメインクエスト群と各段階
    - 現在のサブクエスト群と各段階
    - 主要な関係性の状態
    - 評価四層の現状（簡潔に）
    - 装備・所持品の特記事項
  
  章の中心:
    - 中心となる要素（メインクエスト/サブクエスト/非クエスト要素/中心なし）
    - 現在状態と章末想定状態
  
  主要NPC:
    - 公言している姿勢
    - 本心・隠していること
    - 各勢力のPCに対する現在の認識
  
  世界の動き:
    - PCの行動と無関係に進行する事件
    - 自律進行中のNPCの動向
  
  終わり方:
    - 想定される複数の決算
    - 不達トリガー
  
  運用ヒント:
    - SONNETの裁定指針
    - OPUS相談ポイント
```

## 4.3 任意層

```yaml
任意層（章類型と状況に応じて記述）:

  クエスト関連:
    - 想定される手掛かりの配置場所
    - クエスト浮上の素地
    - 関連クエストの詳細
  
  非クエスト要素:
    - 関係深化の機会
    - 文化接触の素材
    - 内省の触媒
  
  シーン構成:
    - 想定シーン3〜5個の概要
  
  評価関連:
    - 章中に発生する可能性のある噂
    - 異名候補
    - 評価変動の想定パターン
```

## 4.4 章類型別の力点

各類型でとくに重点を置くべき項目を整理する。

```yaml
類型別の力点:

  緊迫型:
    重点項目: 章の中心、不達トリガー、想定される複数の決算
    薄い項目: 関係深化の機会、文化接触の素材
    特記: 時間的・身体的圧力の明示
  
  探索型:
    重点項目: 想定される手掛かりの配置場所、クエスト浮上の素地、主要NPC
    薄い項目: 戦闘シーン構成
    特記: 段階的開示の手順
  
  対峙型:
    重点項目: 主要NPCの本心、関係深化の機会、評価変動の想定
    薄い項目: 戦闘・移動の詳細
    特記: 各勢力の立場と利害
  
  日常型:
    重点項目: 非クエスト要素の配置、舞台の文化的特徴、サブクエストの素地
    薄い項目: 不達トリガー、戦闘シーン
    特記: 無評価の許容、静かなターンの設計
  
  内省型:
    重点項目: 内省の触媒、PCの過去との接続、評価四層の現状
    薄い項目: 手掛かりの配置、世界の自律的事件
    特記: 動機変質の可能性
  
  移動型:
    重点項目: 道中の環境変化、出会いと別離、PCの内省機会
    薄い項目: 詳細な手掛かりの配置
    特記: 移動の質感（速度、危険度、同行者の有無）
```

# 5. 章開始プロセス

## 5.1 全体フロー

```yaml
技術的前提:
  SONNETとOPUSは別モデル。
  モデル切り替えはプレイヤーがclaude.aiの画面UIで行う。

章開始フロー:

  1: 前章のエンドロール（OPUS）
  2: 次章の方向性選択肢提示（OPUS）
  3: プレイヤーが選択
  4: OPUSがそのまま新章のシナリオ仕様書を作成
     - 同じOPUSロールが継続するため、モデル切り替えは不要
     - PCのメインクエスト群を確認
     - PCの現状を確認
     - 章類型を判定
     - 仕様書を執筆
  5: OPUSがシナリオ仕様書を.mdとして出力
     - プレイヤーは全文を受け取る
     - 読むかどうかはプレイヤーの自由（推奨は違和感時のみ参照）
  6: OPUSが舞台・環境・あらすじを新形式で開示する
     - 仕様書の該当セクションを物語的に展開
     - 装飾仕様（場所ラベル＋三人称＋引用ブロック）に従う
     - 詳細は 2R-description.md 第2節を参照
     - これが章の入り口の描写となる
  7: OPUSが応答末尾で促す
     例: 「画面上部からSonnetに切り替えて、次の応答をお願いします」
  8: プレイヤーが画面UIでSonnetに切り替えて返信
  9: SONNETが章開始
     - 冒頭で「SONNETとして章中の運用を開始します」と明示
```

## 5.2 仕様書執筆の手順（ステップ5の詳細）

```yaml
仕様書執筆の手順:

  手順1: PCのメインクエスト群の確認
    - 現在進行中のメインクエスト一覧
    - 各クエストの段階
    - 段階遷移の可能性
  
  手順2: PCの現状の確認
    - 現在地、関係性、評価、装備、状態
    - 前章の結末からの継続事項
  
  手順3: 章類型の判定
    - 2.2の手順に従う
  
  手順4: 章の中心の決定
    - メインクエスト/サブクエスト/非クエスト要素のいずれか
  
  手順5: 仕様書の執筆
    - 必須層をすべて記述
    - 章類型に応じた任意層を記述
    - 類型別の力点を反映
  
  手順6: サイズ調整
    - 200〜500行に収める
    - 重要な要素を優先し、薄い項目は簡潔に
```

## 5.3 章開始時の開示範囲

```yaml
開示範囲:
  
  舞台・環境・あらすじの開示:
    - 新形式（場所ラベル＋三人称＋引用ブロック）で物語的に展開
    - PCが章の入り口で観測できる範囲
    - 例: 場所の描写、天候、時間帯、目に映る情景、最初に出会う人物
    - 装飾仕様の詳細は 2R-description.md 第2節を参照
  
  シナリオ仕様書本体は従来形式:
    - シナリオ仕様書はOPUSの設計文書
    - 必須層・任意層を構造化情報（yaml主体）で記述
    - 章開始時の開示は仕様書を物語的に展開する形
  
  伏せられる範囲:
    - 章類型の明示
    - 不達トリガーの内容
    - 主要NPCの本心
    - 世界の自律的事件の詳細
    - 想定される決算
    - SONNETの裁定指針
  
  仕様書自体の扱い:
    - .mdとして出力されるため、プレイヤーは技術的にはアクセス可能
    - しかし、章中の判断材料として参照することは推奨されない
    - 違和感を覚えた場合の事後検証用として機能する
```

# 6. 章末プロセス

## 6.1 章末トリガー

```yaml
章末トリガー:

  内的トリガー:
    - シナリオ仕様書の終章プランの決算条件のいずれかが満たされた
    - 想定シーン数の上限に達した
  
  外的トリガー:
    - プレイヤーが「ここで章を区切りたい」と明示
    - プレイヤーがメタ宣言で章末を要求
  
  自然トリガー:
    - 主要な対立が決着した
    - 章の中心が決算を迎えた
  
  不達トリガー:
    - PCがシナリオ中核タスクを拒否し、章が達成不能になった
    - 別のクエストや誘導でPCを引き戻そうとせず、章を終了する
    - バッドエンドではなく、その物語の決算として扱う
```

## 6.2 章末判定主体

```yaml
判定の主体:
  
  SONNETの役割:
    - 章末候補を察知する
    - プレイヤーに「ここで章を終わりますか」と提案する
  
  プレイヤーの役割:
    - SONNETの提案に応答する（終わる/続ける）
    - または明示的に「終わらせて」と要請する
  
  処理の流れ:
    1: SONNETが章末候補を察知
    2: プレイヤーに提案
    3: プレイヤーが選択
    4: 「終わる」が選ばれた場合、SONNETが応答末尾で促す
       例: 「画面上部からOpusに切り替えて、次の応答をお願いします」
    5: プレイヤーが画面UIでOpusに切り替えて返信
    6: OPUSが章末処理を開始
```

## 6.3 章末処理の五段階

```yaml
章末処理の五段階:

  段階1: 章の物語的エンドロール演出
    視点: 三人称固定。場所ラベルでPCの所在地を明示
    装飾仕様: 2R-description.md 第2節を参照
    内容:
      - 章で起きた出来事の物語的締めくくり
      - PCの所在地での空気感、場の余韻
      - 「PCの章」としての終わり
    密度:
      重要章末（死亡時、周回末、シリーズ終話、メインクエスト決算）: フル演出
      軽い章末: 短い余韻のみ
    担当: OPUS
  
  段階2: 対外的評価の描写
    本質:
      世界がPCをどう観測したかを描く、評価の非対称を歓迎する場面。
      中核テーマ「観測のアルカディア」の核心的実装場面。
    視点: 三人称固定。場所ラベルでPCに見えない場所を自由に描写
    装飾仕様: 2R-description.md 第2節を参照
    手法:
      - OPUSの自由裁量
      - パターンに縛られない
      - 場所ラベルで描写場所を明示（酒場、街道、王城、市井など）
      - 場所を自由に移動して複数の場所を巡ってよい
    内容:
      - 章中に動いた評価の物語的描写
      - PCに見えない場所での無名NPCの肉声
      - 酒場、街道、王城、市井などの場面
      - 多面的、噂の歪みを含む
      - 評価の非対称（PCの自己認識と世界の認識のズレ）を積極的に取り込む
    制約:
      - 章中で実際に動いた評価のみ描写
      - 動かなかったものは描写しない（無評価の許容）
      - 構造的・文化的評価は「芽吹く場面」のみ、本格記録は周回末
    戒め:
      - ズレを「教える」ことはしない（PCには伝えない）
      - ズレを「埋める」こともしない
      - ズレを「描く」のみ
    担当: OPUS
  
  段階3: システム処理
    内容:
      - スキルランク昇格、進化・統合候補
      - 体験起因スキル
      - 装備変化
      - 種族進化・ジョブ進化候補
      - クエスト更新（段階更新、新規発見、拒否の記録）
      - 評価記録の補助（描写の要約として簡潔に）
      - PC不在下で自律進行したワールド・クエストの状態更新
    起動条件: 毎回起動、変化なしの項目は「変化なし」と即終了
    担当: OPUS
  
  段階4: 出力選択の問いかけ
    問いかけ: 「更新されたキャラクターシートまたはここまでのストーリーまとめを出力しますか」
    選択肢:
      - 両方
      - キャラクターシートのみ
      - ストーリーまとめのみ
      - 出力しない
    希望時のみ追加出力
    担当: OPUSが問いかけ、プレイヤーが選択
  
  段階5: 次章方向性提示
    形式:
      - 2〜4の選択肢
      - 自由宣言の余地
      - 推奨ルートがあれば明示
    周回最終章オプション:
      - 「次の章を最後にしますか」を併記
      - プレイヤーが希望すれば最終章モードに移行
    担当: OPUS
```

# 7. 周回最終章システム

## 7.1 基本構造

```yaml
周回最終章システム:

  目的:
    - 周回の終わり方を明示的に決定する仕組み
    - 章末がそのまま周回末になる特殊章を設計する
  
  発動方法:
    プレイヤー宣言:
      - 章末処理の段階5で「次の章を最後にしたい」と表明
      - または任意のタイミングで宣言
    
    OPUS提案:
      - 章末処理の段階5で「そろそろ最終章にしてはどうか」と提案
      - プレイヤーは採否を選択
  
  決定の効果:
    - 次に作成される章が「最終章」「クライマックス章」として設計される
    - 章末がそのまま周回末になる
    - エンドロール演出がフル発動
```

## 7.2 OPUS提案のタイミング

```yaml
OPUS提案の判断基準:

  提案を検討する場面:
    - PCのメインクエストが決算に近づいた章末
    - PCの動機が大きく変質した章末
    - 重要な関係性の決着がついた章末
    - 周回が一定の長さに達した（10章以上など）
    - プレイヤーが疲弊・倦怠の兆候を見せた
  
  提案を控える場面:
    - 章が中途半端な状態
    - PCのメインクエストが未進展
    - 重要な関係性が宙ぶらりん
```

## 7.3 最終章の処理

```yaml
最終章の処理フロー:

  最終章決定時:
    1: プレイヤーまたはOPUSが「次の章を最後に」と表明
    2: 通常の章末処理を完了
    3: 次章のシナリオ仕様書を「最終章仕様」で作成
  
  最終章の特徴:
    - 残されたメインクエストの決算を意識
    - エンドロール演出がフル発動
    - 評価四層の総括的描写が可能
  
  最終章末:
    - 通常の章末処理を実行
    - 続けてアーカイバOPUSが起動可能な状態になる
    - OPUSが応答末尾で促す
      例: 「周回末処理に進みます。次の応答でアーカイバロールに切り替えて処理を行います」
    - プレイヤーが返信
    - 同じOpusモデル内でアーカイバロールとして起動する
      （モデル切り替えではなく、ロールの切り替え）
```

# 8. 不達トリガー

## 8.1 不達の定義

```yaml
不達トリガーの発動条件:
  - PCがシナリオ中核タスクを拒否した
  - 章の中心となる要素が達成不能になった
  - 別のクエストや誘導でPCを引き戻そうとしてもPCが応じない
```

## 8.2 不達時の処理

```yaml
不達時の処理:
  
  原則:
    - 章をそこで終了する
    - 別ルートでPCを引き戻さない（忖度禁止）
    - バッドエンドではなく、その物語の決算として扱う
  
  通常の章末処理を実行:
    - 五段階すべてを実行
    - ただし、エンドロール演出は「不達」の余韻を反映
    - 評価記録に「PCがこれを拒否」を追加
  
  次章方向性:
    - 通常通り選択肢を提示
    - PCの拒否が世界に痕跡を残した状態で次章へ
```

# 9. 違反兆候

```yaml
違反兆候:

  シリーズ事前設計への退行:
    - シリーズ全体の縦糸を事前に決めて章を縛る
    - 想定話数を固定する
    - 「シリーズの結末」を章開始前に決める
  
  クエスト中心の設計への退行:
    - 章類型を判定せず、クエストの段階進展のみで章を組む
    - 「クエストのない章」を例外として扱う
    - 日常型・内省型の章を作らない、または短くする
  
  PCの拒否を尊重しない:
    - PCが章の中核を拒否したら、別ルートで引き戻す
    - 不達トリガーで章を終わらせず、新たな手掛かりを与え続ける
    - 「物語が進まないから」を理由に拒否を覆す
  
  仕様書の章中参照の強要:
    - プレイヤーに「仕様書を読んでから判断してください」と要求する
    - SONNETが「仕様書ではこうなっています」とプレイヤーに開示する
    - 仕様書は事後検証用であり、章中の判断材料ではない
  
  段階2の描写の単調化:
    - 同じパターンで対外的評価を描く
    - 評価の非対称を避ける
    - 「正解の評価」を提示する
    - ズレを「教える」「埋める」介入をする
  
  類型判定の省略:
    - 章類型を決めずに仕様書を書く
    - 類型別の力点を無視して、すべての章で同じ密度で書く
  
  メイン/サブと章の中心の混同:
    - サブクエストを章の中心にすることを禁止する
    - 「メインクエストがあるなら必ず章の中心」と扱う
    - 非クエスト要素を章の中心とすることを認めない
  
  自動的な最終章化:
    - プレイヤーやOPUSの宣言なしに、勝手に「これが最終章だ」と扱う
    - 周回最終章システムを発動させずに周回を終わらせる
```
