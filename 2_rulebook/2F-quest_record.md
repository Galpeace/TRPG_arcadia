# クエスト記録 様式

## このファイルの位置

クエストの記録フォーマットを定義する。詳細なクエスト運用ルールは `2R-quest.md` を参照。

## 三秒で読む要約

クエストを二軸分類（メイン/サブ × ワールド/ユニーク）で記録。段階管理、自律進行、拒否・放棄も含む。

## 一分で読む要約

クエストは「メイン/サブ × ワールド/ユニーク」の二軸で分類される。本記録はこの分類でクエストを管理し、各クエストの段階（発見・進行・決算）、状態（進行中・完了・拒否・放棄）、関与履歴を記録する。ワールド・クエストはPC不在中の自律進行も記録する。yaml主体で構造化し、必要箇所で散文を使う。拒否・放棄クエストも履歴として記録に残し、PCの選択を尊重する。周回末にOPUSアーカイバが整理し、必要に応じてクエストリスト（6_quest_list/）への更新パッチを生成する。

----

# 1. クエスト記録の基本

```yaml
クエスト記録の基本:
  
  作成・更新主体:
    - キャラメイク時の登録: OPUS設計
    - 章末の更新: OPUS設計（章末処理段階3）
    - 周回末の整理: OPUSアーカイバ
  
  形式:
    - yaml主体＋必要箇所で散文
    - 200〜400行程度（クエスト数による）
  
  位置付け:
    - キャラクターシートの「進行中クエスト」を補完
    - 完全なクエスト履歴を保持
```

----

# 2. クエスト記録の構造

```yaml
クエスト記録の構造:
  
  分類:
    - メイン × ユニーク
    - メイン × ワールド
    - サブ × ユニーク
    - サブ × ワールド
  
  各クエストの情報:
    - ID
    - タイトル
    - 分類
    - 段階
    - 状態
    - 起源
    - 進行履歴
    - 自律進行（ワールド・クエストのみ）
```

# 3. テンプレート

## 3.1 全体構造

```yaml
quest_record:
  
  pc_name: "ヴァランディーネ"
  cycle: 1
  last_updated: "2026-XX-XX"
  
  main_unique:
    - (個別クエスト)
  
  main_world:
    - (個別クエスト)
  
  sub_unique:
    - (個別クエスト)
  
  sub_world:
    - (個別クエスト)
  
  rejected_or_abandoned:
    - (拒否・放棄クエスト)
```

## 3.2 個別クエストのテンプレート

```yaml
- id: "Q-001"
  title: "弟を探す"
  classification: "メイン × ユニーク"
  
  status: "完了"  # 進行中 / 完了 / 拒否 / 放棄
  
  stages:
    - stage: 1
      title: "志を持つ"
      reached_at: "キャラメイク時"
      description: |
        PCの動機「弟を探す」が確立した時点。
        自動発生クエストとして登録。
    
    - stage: 2
      title: "情報収集"
      reached_at: "第1章"
      description: |
        弟の所在地について断片的な情報を集める。
        北方ヨウィン方面に向かったらしい、という噂。
    
    - stage: 3
      title: "手掛かりを掴んだ"
      reached_at: "第2章"
      description: |
        恩師ゴルヴァスから、弟が山賊団に誘拐された情報を得る。
    
    - stage: 4
      title: "山賊団に追跡"
      reached_at: "第3章"
      description: |
        ヨプの大森林を抜けて山賊団に追いついた。
    
    - stage: 5
      title: "救出完了"
      reached_at: "第3章末"
      description: |
        山賊団のリーダーを倒し、弟を救出した。
  
  current_stage: 5
  resolution: "成功"  # 成功 / 失敗 / 中途
  
  origin: |
    自動発生クエスト。
    PCの動機「弟を探す」から派生。
  
  notes: |
    重要なメインクエストとして数章にわたって進行した。
    完了したが、新たなクエスト「家を再興する」（Q-002）が
    主軸となっていく。
```

## 3.3 ワールド・クエストの追加項目

ワールド・クエストには自律進行の項目を追加する。

```yaml
- id: "Q-W-005"
  title: "アルカディア王国とエルフ族の関係"
  classification: "サブ × ワールド"
  
  status: "進行中"
  
  stages:
    - stage: 1
      title: "緊張の発生"
      reached_at: "周回開始前"
      description: |
        アルカディア王国の森林伐採事業がエルフ族との緊張を生む。
    
    - stage: 2
      title: "PCの介入"
      reached_at: "第3章"
      description: |
        PCがエルフ集落の境界を尊重した行動を取った。
        エルフ族の警戒が一部緩和。
  
  current_stage: 2
  
  autonomous_progress:
    - chapter: "第4章間"
      pc_involvement: "なし"
      progress: |
        PCの介入なしに、アルカディア王国側の使節がエルフ集落を訪れた。
        小規模な対話の場が設けられた。
    
    - chapter: "第6章間"
      pc_involvement: "なし"
      progress: |
        森林伐採の縮小が決定された。
        ただしエルフ族の完全な信頼は得られていない。
  
  origin: |
    ワールド・クエストとして章開始時に発見。
    PCの関与は限定的だが、世界としては継続的に動いている。
  
  notes: |
    PCの関与次第で、メインクエスト化する可能性もある。
```

## 3.4 拒否・放棄クエスト

```yaml
rejected_or_abandoned:
  
  - id: "Q-REJ-001"
    title: "アルカディア王国への忠誠"
    classification: "サブ × ユニーク"
    status: "拒否"
    
    proposed_at: "第2章"
    rejected_at: "第2章"
    
    description: |
      アルカディア王国の使者がPCに忠誠を求めた。
      PCは家名の再興を志しているが、現状の王国への
      無条件な忠誠は拒否した。
    
    consequences: |
      アルカディア王国との関係は微妙な状態に。
      ただし敵対関係には至っていない。
  
  - id: "Q-ABN-002"
    title: "酒場の用心棒"
    classification: "サブ × ユニーク"
    status: "放棄"
    
    accepted_at: "第1章"
    abandoned_at: "第2章"
    
    description: |
      北方ヨウィンの酒場での用心棒の仕事を引き受けた。
      しかし弟の手掛かりを得て、優先順位を変えた。
    
    consequences: |
      酒場の主人との関係が少し悪化。
      ただし金銭的な補償をしたため、決定的な決裂はない。
```

# 4. 段階管理の指針

## 4.1 段階の粒度

```yaml
段階の粒度:
  
  原則:
    クエストごとに3〜5段階で管理する。
    細かすぎず、粗すぎず。
  
  典型的な段階:
    1: 発見・志を持つ
    2: 情報収集・準備
    3: 手掛かり・接触
    4: 直接的な行動・対峙
    5: 決算（完了・失敗）
  
  バリエーション:
    - 長期クエストはさらに細分化（10段階程度まで）
    - 短期クエストは2〜3段階で十分
```

## 4.2 段階の到達タイミング

```yaml
段階の到達タイミング:
  
  原則:
    各段階に到達したタイミング（章番号）を記録する。
    プレイヤーが「いつ何が起きたか」を振り返れるようにする。
  
  記録例:
    reached_at: "第3章"
    reached_at: "キャラメイク時"
    reached_at: "周回開始前"  # 自律進行で既に始まっていた場合
```

# 5. クエストの状態

```yaml
状態の分類:
  
  進行中:
    PCが関与している、または関与の可能性がある状態
  
  完了:
    決算が成功した状態
  
  失敗:
    決算が失敗した状態
  
  拒否:
    PCが明示的に拒否した状態
    別ルートでの誘導はしない
  
  放棄:
    PCが一度引き受けたが、後に放棄した状態
  
  保留:
    一時的に動いていないが、再開可能な状態

状態の遷移:
  
  進行中 → 完了 / 失敗 / 放棄
  保留 → 進行中
  進行中 → 保留
  
  拒否は最終状態（再開しない）
```

# 6. テンプレートの運用

## 6.1 厳密性

```yaml
フォーマットの厳密性:
  
  必須項目:
    - id, title, classification, status
    - 少なくとも1つの段階
    - current_stage
  
  任意項目:
    - stages の詳細
    - autonomous_progress（ワールド・クエストのみ必須）
    - notes
    - resolution（完了時のみ必須）
  
  変形:
    - クエストの性質に応じて項目を追加可能
    - 長期クエストは段階を細分化
    - 短期クエストは簡潔に
```

## 6.2 散文と構造化情報

```yaml
使い分け:
  
  散文を使う箇所:
    - 各段階の description
    - notes
    - origin
    - autonomous_progress の progress
    - consequences
  
  構造化情報のままの箇所:
    - id, title, classification, status
    - stage 番号
    - reached_at
    - resolution
```

## 6.3 周回をまたぐ場合

```yaml
周回をまたぐクエスト:
  
  原則:
    ワールド・クエストの一部は、周回をまたいで継続する。
    その場合、cycle 1, cycle 2, cycle 3 と分けて記録する。
  
  記録例:
    autonomous_progress:
      - cycle: 1
        chapter: "周回末以降"
        progress: |
          (cycle 1 終了後の進行内容)
      
      - cycle: 2
        chapter: "新規開始時"
        progress: |
          (cycle 2 開始時点の状態)
  
  注意:
    - PCが別人格になっても、ワールド・クエストは継続することがある
    - 旧PCが関わっていたクエストの「歴史」は残る
```

# 7. 違反兆候

```yaml
違反兆候:

  記録の不徹底:
    - クエストを章末で更新しない
    - 自律進行を記録しない
    - 拒否・放棄を「なかったこと」にする
  
  段階管理の混乱:
    - 段階を細分化しすぎる
    - 段階を粗くしすぎる
    - 「いつ何が起きたか」を記録しない
  
  分類の混乱:
    - メインとサブの判別を曖昧にする
    - ワールドとユニークの区別を無視する
    - 「重要度の変化」を反映しない
  
  拒否・放棄の歪曲:
    - 拒否されたクエストを「別ルート」で再提示
    - 放棄されたクエストを「結局やらせる」展開
    - PCの選択を尊重しない
  
  キャラシートとの責任の混乱:
    - キャラシートにクエストの詳細履歴を書く
    - 本記録にキャラシートの基本情報を重複記載
  
  自律進行の機械化:
    - ワールド・クエストの自律進行を毎章機械的に進める
    - PCの関与の有無を反映しない
    - 「進行のための進行」になる
```
