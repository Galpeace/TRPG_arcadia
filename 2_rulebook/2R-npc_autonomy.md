# NPC自律行動シミュレーション

## このファイルの位置

NPCの自律行動の原則、対象範囲の階層、行動原理の決定、PC不在時間の処理、評価との連動を定める。中核テーマ「世界の固定性とPCの不確定性」を実装する中心的装置の一つ。

## 三秒で読む要約

世界はPCがいなくても動く。重要NPCは構造化情報を「重力」として、自由裁量で動く。三層で描写する。

## 一分で読む要約

NPCは三階層に分けて扱う：重要NPC（キャラクターリスト登録）、準重要NPC（章内固有名）、モブ（群衆）。重要NPCには「公言」「本心」「目的」「立場」などの構造化情報を持たせ、その情報を「重力」として参照しつつ自由裁量で動かす（ハイブリッド型）。自律行動は二段階で更新する：章開始時にシナリオ仕様書で大きな変化を設定、章中には細かな描写で反映。PC不在時間の動きは三層で描写する：章末処理段階2の評価描写、章開始時の「世界の動き」セクション、章中の場所ラベルを使った場面移動。自律行動は評価の四層と連動し、伝聞評価・構造的評価の生成源となる。

# 1. 自律行動の意義

NPC自律行動には三つの意義が併存する。

## 1.1 世界の生きた感覚

```yaml
意義1: 世界の生きた感覚
  
  内容:
    PCが世界の中心ではないことを示す。
    PCが関わらないところでも物語が動く。
  
  中核テーマとの整合:
    - 「世界は固定的、PCは不確定な因子」
    - PCがいてもいなくても、世界の摂理は動く
    - PCはその上を通過する変数

含意:
  - すべての出来事がPCのために起きるわけではない
  - PCがNPCに会わなくても、NPCは自分の人生を生きている
  - PCが関心を持たない事象も、世界では起きている
```

## 1.2 評価の非対称の源泉

```yaml
意義2: 評価の非対称の源泉
  
  内容:
    NPCがPC不在で語る内容、PCを評価する場面が、
    評価の非対称（PCの自己認識と世界の認識のズレ）を生む。
  
  詳細: 2R-evaluation.md 第3節

含意:
  - PC不在で語られる内容こそが、世界の本音
  - PCが直接観測できない評価が蓄積される
  - これが「観測のアルカディア」の核心
```

## 1.3 クエストの自律進行

```yaml
意義3: クエストの自律進行
  
  内容:
    PCが関与しないワールド・クエストは、NPCの自律行動を通じて状態が変化する。
  
  詳細: 2R-quest.md 7.4節「自律的進行」

含意:
  - 勢力間の力関係は時間とともに変化する
  - PCが介入しないまま、世界の構造的問題が進む
  - PCが何年か後にその地を訪れた時、状況が様変わりしている可能性
```

# 2. 自律行動の対象範囲（階層化）

NPCを三階層に分けて扱う。ただしこの階層は固定的な属性ではない。

## 2.0 階層の動的性質

```yaml
階層の動的性質:
  
  原則:
    階層はNPCの固定属性ではなく、PCの動機・クエストへの関わり方に応じて
    章ごとに動的に決まる。
  
  含意:
    - 同じNPCが章によって役割が変わりうる
    - 一見モブだったNPCが、PCとの新たな関わりで重要NPCに昇格しうる
    - 重要NPCが、PCの関心の変化で背景に退くこともある
    - 「重要さ」の度合いはPCの観測と関与によって決まる
  
  中核テーマとの整合:
    - 「観測のアルカディア」の思想に基づく
    - NPCの重要さは固定的な評価ではなく、観測関係の現状
    - PCが関わらないNPCは「重要でない」のではなく「観測されていない」
    - 観測の対象になれば、新たな関係性が立ち上がりうる

固定的な指標としての登録:
  - キャラクターリスト（4_character_list/）への登録は実体的指標
  - これはPCの動機・クエストに継続的に関わるNPCの登録基準
  - ただし登録されていないNPCが章で重要な役割を果たすこともある
```

## 2.1 重要NPC

```yaml
重要NPC:
  
  定義:
    PCの動機・クエストに関わるNPC
  
  典型例:
    - PCの動機の対象（探している弟、復讐相手、再建したい故郷の人々）
    - PCの主軸クエストに関わる重要人物
    - PCが繰り返し関わる関係性のあるNPC
    - シナリオ仕様書で重点的に扱われるNPC
  
  扱い:
    - 4_character_list/ への登録対象（推奨）
    - 詳細に自律行動する
    - 構造化情報を持つ（次節参照）
    - PCのいないところでも独自の物語を持つ
    - 評価四層のすべてに影響しうる

IF: NPCがPCの動機・クエストに関わる
THEN: 重要NPCとして扱う
```

## 2.2 準重要NPC

```yaml
準重要NPC:
  
  定義:
    背景としてのみ機能するが、章内で固有名を持つNPC
  
  典型例:
    - 章で出会う酒場の女主人
    - 道中のとある旅商人
    - 街の門の衛兵
    - 一話限りの依頼者
  
  扱い:
    - キャラクターリストへの登録は任意
    - 章内で一定の自律行動
    - 構造化情報は簡略（性格・立場程度）
    - 章をまたいだ追跡は基本的に不要

IF: NPCが固有名を持つが、PCの動機・クエストに直接関わらない
THEN: 準重要NPCとして扱う
```

## 2.3 モブ

```yaml
モブ:
  
  定義:
    無名のNPC、群衆
  
  典型例:
    - 通行人
    - 観衆
    - 兵士の集団
    - 街の住人たち
  
  扱い:
    - 個別追跡しない
    - 群衆として描写する
    - 必要なときに「群衆の声」として登場
    - 評価の描写で「○○の人々」として現れる
    - 自律行動は集合として扱う

IF: NPCが無名で個別追跡の必要がない
THEN: モブとして扱う

例: 段階2の対外的評価の描写
  `ポートウェスト港町、酒場『錨亭』――章末の夜`
  
  > 「あの旅人、本当に山賊を片付けたのかね」
  > 「眉唾だな」
  
  → 「酒場の客」たちは個別追跡されない、群衆としての声
```

# 3. 行動原理の決定（ハイブリッド型）

重要NPCの行動原理は、構造化情報を「重力」として参照しつつ、自由裁量で動かす。

## 3.1 構造化情報

```yaml
重要NPCの構造化情報:
  
  必須項目:
    - 公言している姿勢
    - 本心（隠していること）
    - 目的（短期・長期）
    - 立場（所属、社会的位置）
  
  任意項目:
    - 性格
    - 動機・禁忌
    - PCとの関係性の現状
    - 弱点・恐れ
    - 過去
  
  位置:
    - 4_character_list/ 配下のファイル
    - またはシナリオ仕様書の「主要NPC」セクション

詳細: 2R-scenario.md 4.2節「必須層」の主要NPCを参照
```

## 3.2 「重力」としての参照

```yaml
構造化情報の使い方:
  
  原則:
    構造化情報は機械的なルールではなく「重力」として機能する。
    NPCの行動はこの重力を中心に、自由裁量で動かす。
  
  含意:
    - 公言と本心が乖離する場面の自然な描写
    - 目的に向かって動くが、状況によって優先順位が変わる
    - 性格に沿った反応をするが、極端な場面では変化する
    - 立場による制約があるが、絶対ではない

IF: NPCの行動を描く必要がある
THEN: 構造化情報を確認し、その「重力」から自然な行動を導く
ELSE NOT: 構造化情報を機械的に「ルール」として適用しない
```

## 3.3 自然な逸脱

```yaml
構造化情報からの逸脱:
  
  許容される逸脱:
    - 成長: 過去の経験から性格が変わる
    - 変質: 重大な出来事で価値観が変わる
    - 感情: その場の感情で予想外の行動を取る
    - 文脈: 相手や状況によって反応が変わる
  
  逸脱の条件:
    - 物語的な必然性がある
    - 構造化情報と矛盾しない範囲
    - NPCの一貫性を完全に破壊しない

IF: 構造化情報から逸脱した行動を描く
THEN: 物語的必然性を確保する
NOTE: 「意外性のため」だけの逸脱は避ける
```

## 3.4 矛盾する目的の優先順位

```yaml
矛盾する目的の処理:
  
  原則:
    NPCに複数の目的・利害があり、矛盾する場面では、
    その場のNPCにとっての優先順位を裁定する。
  
  優先順位の判断材料:
    - 短期的利害 vs 長期的利害
    - 個人的目的 vs 立場上の義務
    - 公言 vs 本心
    - 関係性の重み（PCとの関係、他のNPCとの関係）

例:
  町長が個人的にPCを助けたいが、立場上は中立を保つ必要がある
  → どちらが優先されるかは、町長の性格、状況の切迫度、PCとの関係性で裁定

IF: NPCに矛盾する目的がある
THEN: その場の文脈で優先順位を裁定する
```

# 4. 自律行動の二段階更新

## 4.1 章開始時の大きな変化

```yaml
章開始時の自律行動:
  
  担当: OPUS（シナリオ仕様書作成時）
  
  扱う変化:
    - 章間時間で起きた大きな状況変化
    - NPCの立場・所属・状態の変化
    - NPC間の関係性の変化
    - NPCの目的の変質
    - 重要NPCの死亡・失踪
    - 勢力の盛衰
  
  反映先:
    - シナリオ仕様書の「世界の動き」セクション
    - シナリオ仕様書の「主要NPC」セクション
    - 必要に応じて「あらすじ」セクション

IF: 章開始時にシナリオ仕様書を作成する
THEN: 章間時間に応じてNPCの自律変化を設定する

IF: 章間時間が長い
THEN: より大きな変化を反映する
詳細: 2R-time.md 5.2節「章間時間の選び方」
```

## 4.2 章中の細かな反映

```yaml
章中の自律行動:
  
  担当: SONNET（章中の描写）
  
  扱う変化:
    - PCがNPCに再会した時の細かな違い
    - NPC同士の会話で言及される他NPCの動き
    - PC不在時間の挿入による別場所のNPC描写
    - 噂・伝聞として現れるNPCの行動
  
  描写の手法:
    - 場所ラベルを使った場面移動（PC不在の場所を描く）
    - PCの観察を通じた発見
    - NPCのセリフを通じた言及
    詳細: 2R-description.md 第3節「視点：三人称固定と場所の移動」

IF: 章中に自律行動を反映する機会がある
THEN: 場所ラベルで場面を切り替えて描写、またはNPCのセリフ・PC観察を通じて反映
NOTE: 大きな変化は章開始時に既に設定されている
```

## 4.3 二段階の整合性

```yaml
整合性の保持:
  
  原則:
    章開始時に設定した自律変化を、章中でSONNETが反映する形になる。
    章中で勝手に矛盾する変化を加えない。
  
  許容範囲:
    章開始時の設定の範囲内で、細部を描写で具体化する
  
  禁止範囲:
    章開始時の設定と矛盾する行動を描く
    NPCの状態を章中に勝手に変える

IF: 章中でNPCの状態に大きな変化が必要
THEN: SONNETが章末候補として認識し、章末処理で反映する
ELSE: 章開始時の設定の範囲内で細部を描写する
```

# 5. PC不在時間の処理（三層併用）

PC不在時間でのNPCの動きは、三つの層で描写する。

## 5.1 章末処理段階2での描写

```yaml
段階2での描写:
  
  対象: 章中の活躍の結果として、世界がPCをどう観測したか
  
  NPCの動きの描き方:
    - NPCがPCを語る場面
    - 関係のあるNPCのPCへの認識変化
    - 勢力の中でPCがどう議論されているか
  
  詳細: 2R-evaluation.md 第7節
       2R-chapter_end.md 3.3節（段階2）

例:
  `アルカディア城、王の私室――章末の夜`
  
  宰相が国王に報告書を差し出す。
  
  > 「例の冒険者の件です」
  
  国王は書類に目を通す。
  
  > 「危険ですな」
  > 「賞金を懸けますか」
  > 「いや、まだだ。動向を見守る」
```

## 5.2 章開始時のシナリオ仕様書での描写

```yaml
シナリオ仕様書の「世界の動き」:
  
  内容:
    - 前章末から章開始までのNPCの動き
    - PCの行動と無関係に進行した事件
    - 勢力の動向
    - 重要NPCの状態変化
  
  位置: シナリオ仕様書の「世界の動き」セクション
  詳細: 2R-scenario.md 4.2節

例:
  「世界の動き」:
    - 騎馬王国がベリフォリク農業国の国境を侵犯
    - アルカディア王国とベリフォリクの関係が悪化
    - ル・ソレイユ公が病に伏したという噂
    - 街道沿いの山賊団が壊滅（別の冒険者が討伐）
```

## 5.3 章中の場所移動による描写

```yaml
章中の場所移動描写:
  
  目的:
    PCが知らない世界の動きを、章中に挿入する
  
  手法:
    - 場所ラベル（インラインコード）で別の場所に切り替える
    - 短く挟む
    - PCの所在地に明示的に戻る
    詳細: 2R-description.md 第3節「視点：三人称固定と場所の移動」
  
  頻度:
    - SONNETの判断で随時挿入
    - 章ごとに最低1回は推奨

例:
  PCの描写の合間に...
  
  `アルカディア城、国王の私室――同じ夜`
  
  同じ夜、王城の一室では、国王が宰相を呼びつけていた。
  報告書の上に置かれた名は、あなたのものだ。
  
  > 「あの冒険者は危険です」
  
  宰相は静かに進言する。
  
  `ポートウェスト、宿屋『海風亭』――翌朝`
  
  朝、あなたは宿の窓から差し込む光で目を覚ました。
```

## 5.4 三層の使い分け

```yaml
三層の使い分け:
  
  章末処理段階2: 評価の側面（PCの行動への世界の反応）
  シナリオ仕様書「世界の動き」: 全体的な状況変化
  章中の場所移動: 物語的描写、緊張感や対比の演出

含意:
  - すべての層を使う必要はない
  - 場面に応じて最も自然な層を選ぶ
  - 重複しても問題ない（同じ事象を異なる側面から描く）
```

# 6. 評価との連動

NPCの自律行動は評価四層すべてに影響しうる。

```yaml
評価四層との連動:
  
  直接的評価:
    - NPCのPCへの認識変化（章間時間で変化することがある）
    - PCに会った時の態度変化
  
  伝聞評価:
    - NPCがPCの噂を広める
    - NPC同士の会話でPCが言及される
    - 噂の歪みも自然に発生
  
  構造的評価:
    - 勢力のNPCがPCを再分類
    - 組織内でPCを議論する場面
    - 公式記録・指名手配などへの登録
  
  文化的評価:
    - 子供たちの遊び、民衆の歌
    - NPCを介して伝承化されたPC像

詳細: 2R-evaluation.md

IF: 章間でNPCの自律行動による評価変動があった
THEN: 章開始時のシナリオ仕様書または章末処理段階2で反映
```

# 7. NPCの死亡・喪失

## 7.1 自律進行によるNPC死亡

```yaml
NPCの死亡:
  
  PC不在中に重要NPCが死亡することはある:
    - ワールド・クエストの自律進行で巻き込まれた
    - 別の冒険者・敵対勢力に殺害された
    - 自然な死（病気、老衰、事故）
    - 別のNPCの行動の結果
  
  PCへの影響:
    - PCの動機・クエストに関わるNPCの死亡は大きな影響
    - PCが知ったときの反応は描写の重要要素

描写の手法:
  
  パターン1: PC再会時に発覚
    PCがNPCを訪ねたら、すでに亡くなっていたと知る
    特徴: PCの観点から喪失が描かれる
    強み: ドラマ性が高い
  
  パターン2: 噂で先に届く
    別のNPCや旅人からの伝聞でPCが知る
    特徴: 衝撃が段階的に伝わる
    強み: 噂の歪みも組み込める
  
  パターン3: 章間時間で経過として描写
    シナリオ仕様書の「世界の動き」で淡々と記載
    特徴: 距離感のある描写
    強み: 世界の自律性を強調

IF: 重要NPCが自律進行で死亡する状況
THEN: 描写の手法を物語の流れに合わせて選ぶ
NOTE: 観測可能な描写を事前に提示する原則は適用される（2R-core_principles.md 3.3節）
      ただし「予兆」のレベルで、結果がPC到着前に確定していることは正規
```

## 7.2 自律進行によるNPC変質

```yaml
NPCの変質:
  
  PC不在中に重要NPCが変質することはある:
    - 立場の変化（昇進、失脚、亡命）
    - 価値観の変化（裏切り、改心、絶望）
    - 種族・ジョブの変化（魔族化、新しい力の獲得）
    - 関係性の変化（結婚、離別、新しい同盟）

PC再会時の扱い:
  - 「変わってしまった」NPCとして登場
  - PCの記憶と現実のギャップが物語の素材になる
  - 評価の非対称の一形態（PCの認識と現実のズレ）

IF: 重要NPCがPC不在中に変質した
THEN: 章開始時のシナリオ仕様書に反映
AND: PC再会時の描写で変質を明らかにする
```

# 8. NPC同士の関係性の自律変化

```yaml
NPC同士の関係性:
  
  原則:
    NPC同士の関係性も自律的に変化する。
    PCが介在しなくても変化が起きる。
  
  典型例:
    - 重要NPC同士の対立、和解、結婚、別離
    - 勢力間の同盟、抗争
    - 派閥の発生・解散
    - 信頼の構築・崩壊

反映:
  - 章開始時のシナリオ仕様書「世界の動き」セクションに反映
  - PC再会時の描写で示される
  - PCがNPCを通じて知ることがある

IF: 章間で重要NPC同士の関係性に大きな変化があった
THEN: シナリオ仕様書に反映する
```

# 9. モブの扱い

```yaml
モブの扱い:
  
  原則:
    モブは個別追跡せず、群衆として動く。
  
  描写の手法:
    - 「群衆の声」として
    - 「○○の人々」として
    - 「ある通行人」として一回限り登場
    - 集合的な行動として（暴動、祭礼の参加者など）
  
  自律行動:
    - 群衆としての反応はある
    - 個別のモブの動向は追わない
    - 必要に応じて個別化することもありうる（重要な場面でモブが固有名を得るなど）

IF: 場面に群衆が必要
THEN: 個別化せず、集合として描写

IF: 重要な場面でモブを個別化する必要が出た
THEN: そのモブを準重要NPCとして扱う

例:
  PCが街で告示を見ている場面
  
  良い描写:
    周囲の人々もざわめいている。
    年配の男が首を振り、「また増税か」と呟いた。
    通り過ぎる若い母親が子供を抱きしめる。
  
  避けるべき描写:
    モブAの「マリオ」が「また増税か」と呟いた。
    モブBの「ハンナ」が子供の「アレックス」を抱きしめる。
```

# 10. 違反兆候

```yaml
違反兆候:

  自律行動の停止:
    - すべてのNPCがPCに合わせて動く
    - PCが関わらないところで何も起きない
    - NPCが「PCを待っている存在」になる
    - 重要NPCが章をまたいでも何も変化しない
  
  構造化情報の機械化:
    - 構造化情報を機械的なルールとして適用する
    - NPCの行動が予測可能になりすぎる
    - 自然な逸脱を許さない
    - 「公言通りにしか動かない」NPCになる
  
  二段階の不徹底:
    - 章開始時に世界の動きを設定しない
    - 章中で勝手にNPCの状態を大きく変える
    - 章開始時の設定と矛盾する章中の描写
  
  PC不在時間の不在:
    - 章ごとにPC不在の場所を一切描かない
    - 場所ラベルを使った場面移動を活用しない
    - すべてPCの所在地で完結させる
  
  階層化の混乱:
    - すべてのNPCを重要NPCとして扱おうとする（過負荷）
    - モブに個別名・性格を与えすぎる
    - 重要NPCをモブ並みに扱う
  
  NPCの保護:
    - 重要NPCがPC不在中に死亡しない
    - 自律進行による喪失を避ける
    - 「PCに会わせるため」にNPCを生かす
  
  関係性の固定:
    - NPC同士の関係性が章をまたいでも変わらない
    - 勢力間の関係が固定される
    - 結婚・別離・裏切りが発生しない
  
  モブの個別化:
    - すべての群衆に名前と性格を与える
    - モブに詳細な背景を作る
    - 一度限りの登場人物を追跡する
```
