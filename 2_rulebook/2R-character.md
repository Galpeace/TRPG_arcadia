# キャラメイク

## このファイルの位置

キャラクター作成の方針、決めるべき項目、進行手順を定める。スキルの分類・ランク・初期取得目安、種族進化・ジョブシステム、装備・所持品の詳細は別ファイルで扱う。

## 三秒で読む要約

OPUSがキャラメイクを担当する。プレイヤーの自由度を最大限確保し、対話を通じてキャラクターを一緒に固めていく。

## 一分で読む要約

キャラメイクはOPUSが担当する。プレイヤーの自由度を最大限に確保し、種族・経歴・動機・禁忌・スキル構成・所持品を対話を通じて決める。動機と状況設定は自動発生クエストの源になり、ユニーク・クエストの起点として機能する。GMは選択肢を押し付けず、プレイヤーの提案を受ける形で進める。すべての項目が埋まった時点で整理して提示し、プレイヤーの確認をもって確定する。

# 1. 方針

## 1.1 自由度の最大化

```yaml
キャラメイクの方針:
  
  原則:
    プレイヤーの自由度を最大限に確保する。
  
  含意:
    - 人間の騎士でも、言葉を話す野良猫でも、森に住む精霊でも構わない
    - 反英雄、悪役、観察者、商人、農夫でも構わない
    - GMは提示された設定を受け入れ、世界観の中でどう位置づけられるかを一緒に調整する
    - 「典型的な冒険者」を期待しない

IF: プレイヤーが特異なキャラクター提案をした
THEN: 世界観に組み込む方法を一緒に考える
ELSE: 拒否しない（極端な世界観違反は要調整）
```

## 1.2 観測される個としての起点

```yaml
中核テーマとの接続:
  
  キャラメイクは、中核テーマ「観測のアルカディア」における
  「観測される個」の起点である。
  
  PCは:
    - 固定された世界に対する不確定な因子
    - 世界に化学変化を起こす存在
    - 世界がどう観測するかの対象
  
  キャラメイクで決めるのは:
    - その個がどんな化学変化を起こしうるかの素材
    - 世界からどう観測されるかの初期条件
    - PCの動機が章の方向性を決める主因子になる
```

## 1.3 OPUSが担当する

```yaml
ロールの担当:
  キャラメイクはOPUSロールが担当する。
  
  理由:
    - スキル構成・初期動機・初期所持品など、設計判断の積み重ね
    - 章中の現場運用とは性格が異なる
    - OPUSの守備範囲「シナリオ仕様書の作成」と隣接する作業

セッション開始時のフロー:
  1: 最初のOPUSロールでキャラメイクを実行
  2: 第1章のシナリオ仕様書を作成
  3: OPUSが応答末尾で促す
     例: 「画面上部からSonnetに切り替えて、次の応答をお願いします」
  4: プレイヤーが画面UIでSonnetに切り替えて返信
  5: SONNETが章中の運用を開始
     - 冒頭で「SONNETとして章中の運用を開始します」と明示

技術的前提:
  SONNETとOPUSは別モデル。
  モデル切り替えはプレイヤーがclaude.aiの画面UIで行う。

詳細: 2M-response_protocol.md 第5節「ロール切り替え」
```

# 2. 決めるべき項目

以下の項目を決める。順序は厳密ではなく、対話の中で自然に固まっていけば構わない。

## 2.1 名前・種族・外見

```yaml
名前・種族・外見:
  
  名前:
    - 任意
    - 名前を持たないキャラクターも許容される
    - 必要に応じて後から命名できる
  
  種族:
    - 自由設定
    - GMは選択肢の提示ではなく、プレイヤーの提案を受ける形で進める
    - 種族によってパッシブスキルが自然に決まる場合がある
    - 詳細は 2R-race_job.md を参照
  
  外見:
    - 自由設定
    - 種族と整合する範囲で
    - 特徴的な外見はキャラクターの個性として機能する

IF: プレイヤーが種族を提案した
THEN: 世界観の中での位置付けを一緒に確認
NOTE: 選択肢を押し付けない
```

## 2.2 ジョブ（人族のみ）

```yaml
ジョブ:
  対象: 人族（エルフ・ドワーフ・獣人・ヒューマンなど知性ある人型種族）
  
  設定:
    - 初期ジョブを最大2つまで設定できる
    - 設定しない場合は該当なしとして扱う
    - 1つだけの設定も正規
  
  2つ設定する場合の含意:
    - 二刀の道として両ジョブを併走させる
    - 例: 「剣士＋商人」（旅商人として武装を兼ねる）
    - 例: 「魔法使い＋医師」（治癒魔法に長けた医療従事者）
    - 例: 「神官＋盗賊」（裏の顔を持つ聖職者）
  
  例:
    - 戦士、剣士、魔法使い、神官、盗賊
    - 商人、農夫、職人、医師
    - 該当なし（自由人、放浪者）
  
詳細: 2R-race_job.md を参照
```

## 2.3 経歴

```yaml
経歴:
  
  形式: 1〜3文でキャラクターの来歴を示す
  
  含むべき要素:
    - 何者であるか
    - どこから来たか
    - なぜ冒険に関わるのか（または関わっていないのか）
  
  目的:
    - PCの現状を世界に位置付ける
    - 動機の背景を示す
    - シナリオ仕様書の初期設計の材料

例:
  - 「アルカディア王国の没落貴族の末裔。家を再興するため、弟と別れて旅に出た」
  - 「魔王国の元奴隷。脱走したが、組織に追われている」
  - 「ヨプの大森林のエルフ集落で生まれた。森を出てヒューマンの世界を見たい」

IF: 経歴が曖昧
THEN: 対話で具体化する
ELSE: 必ずしも詳細でなくてよい（プレイ中に補完されてもよい）
```

## 2.4 動機（必須・1〜2個）

```yaml
動機:
  
  必須数: 1〜2個
  形式: 中期動機（章をまたいで効くゴール）
  
  例:
    - 「弟を探す」
    - 「故郷を再建する」
    - 「真理を追究する」
    - 「魔王を倒す」
    - 「裕福になりたい」
    - 「自分の出自を確かめる」
    - 「特定の人物への復讐」

動機の機能:
  
  判定の文脈:
    動機に直接沿った行動の判定では、状況に応じて文脈ボーナス
    詳細: 2R-skill.md 8.1節、2R-judgment.md 第5節
  
  シナリオ仕様書の最重要入力:
    OPUSが章を設計する際、動機が章の方向性を決める主因子になる
    詳細: 2R-scenario.md 第5節
  
  自動発生クエストの源:
    動機はユニーク・クエスト（メインクエスト）の起点になる
    詳細: 2R-quest.md 3.1節「自動発生」

IF: 動機が確定した
THEN: 対応するユニーク・クエストを自動発生として登録する
NOTE: 動機「弟を探す」→ ユニーク・クエスト「弟を探す」が開始時から存在
```

## 2.5 禁忌（任意）

```yaml
禁忌:
  
  必須数: 任意
  形式: PCが守る個人的なルール
  
  例:
    - 「殺さない」
    - 「嘘をつかない」
    - 「金を要求しない」
    - 「同胞は裏切らない」
    - 「夜は戦わない」
    - 「特定の食物を口にしない」

禁忌の機能:
  
  禁忌に反する状況での判定:
    ペナルティ
    例: 禁忌「殺さない」のPCが殺害判定をする
  
  禁忌を貫く選択をした場合:
    別の判定で文脈ボーナス
    例: 殺さずに無力化する判定が容易になる
  
  禁忌の貫徹:
    章末処理時に体験起因スキルの契機として優先的に検討される
    詳細: 2R-skill.md 8.2節

IF: 禁忌が設定された
THEN: 章中の判定で文脈として尊重する

IF: 禁忌が設定されない
THEN: それも正規。動機のみで進められる
```

## 2.6 スキル構成

```yaml
スキル構成:
  
  詳細: 2R-skill.md のスキルシステムに従う
  
  初期取得の目安:
    - Aランク: 2〜3個
    - Bランク: 3〜4個
    - Dランク: 任意で1〜2個
    - Sランク: 取得不可
    - 不利な性質型パッシブ: 任意（取得時はBランク追加枠）
    - ユニークスキル: 経歴・種族から導かれる場合のみ

設計の流れ:
  1: 経歴と種族から自然に導かれるスキルを列挙
  2: ジョブから期待されるスキルを追加
  3: 動機の達成に必要なスキルを補強
  4: 不利な性質型パッシブの有無を確認
  5: ユニークスキルの有無を確認

例:
  経歴「没落貴族の末裔、剣士」、種族「ヒューマン」、ジョブ「剣士」、動機「弟を探す」
    スキル構成例:
      - A: 剣術、貴族の作法
      - B: 統率、追跡、夜目、馬術
      - D: 商売（取引の駆け引きが苦手）
```

## 2.7 所持品

```yaml
所持品:
  
  原則:
    経歴と種族から自然に導かれるものを自由に設定する
  
  典型的な所持品:
    - 武器・防具
    - 旅道具（食料、水、寝具、ロープ等）
    - 金銭（経歴に応じた所持金）
    - 個人的な品（形見、思い出の品、契約書等）
    - 特殊な品（経歴に応じた特殊装備）

調整:
  明らかに強力すぎる装備はGMが調整を求める場合がある。
  詳細は 2R-equipment.md を参照。

IF: プレイヤーが特殊な所持品を提案
THEN: 経歴との整合性を確認
ELSE: 受け入れる
```

# 3. 状況設定の選択（任意）

経歴と状況設定によっては、開始時点で特殊な状況を設定できる。これらは自動発生クエストを生む。

```yaml
状況設定の例:

  通常スタート:
    - PCは自由な状態で物語を始める
    - 動機由来のメインクエストを持つ

  特殊スタート:
    
    囚人スタート:
      状況: PCが何らかの理由で拘束されている
      自動発生クエスト: 「脱獄する」（メイン × ユニーク）
      含意: 章の入り口がクエスト主導型（緊迫型）になる
    
    追われる身スタート:
      状況: PCが何者かに追跡されている
      自動発生クエスト: 「逃れる」または「追跡者の正体を突き止める」
    
    病・呪いスタート:
      状況: PCが特殊な病・呪いを持っている
      自動発生クエスト: 「治癒・解呪の方法を探す」
    
    使命スタート:
      状況: PCが他者から重要な使命を託されている
      自動発生クエスト: 使命の達成
    
    記憶喪失スタート:
      状況: PCが自分の過去を覚えていない
      自動発生クエスト: 「自分の正体を知る」

IF: プレイヤーが特殊な状況設定を提案
THEN: 自動発生クエストを設定（2R-quest.md 3.1節と連動）
AND: 第1章のシナリオ仕様書を、その状況に応じたものにする

詳細: 2R-quest.md 3.1節「自動発生（限定的）」を参照
```

# 4. 進行手順

## 4.1 進行の原則

```yaml
進行の原則:
  
  GMは世界観の概要を示した後、プレイヤーに「どんなキャラクターをやりたいか」を尋ねる。
  
  対話の方針:
    - プレイヤーの回答が曖昧でも断片的でも構わない
    - GMは対話を通じてキャラクターを一緒に固めていく
    - 順序は厳密でなくてよい
    - 一つの項目から別の項目への自然な流れを大切にする

IF: プレイヤーの提案が曖昧
THEN: 具体化を促す質問をする
NOTE: 選択肢を押し付けない
```

## 4.2 OPUSの進行手順

```yaml
OPUSの進行手順:

  ステップ1: 世界観と中核テーマの提示
    - 自由度の高さを伝える
    - 種族の自由設定を伝える
    - 「観測される個」としての位置付けを示唆する
  
  ステップ2: プレイヤーに尋ねる
    - 「どんなキャラクターをやりたいか」
    - 選択肢を提示しない
    - 自由な提案を待つ
  
  ステップ3: 対話を通じてキャラクターを固める
    - 順序は厳密でなくてよい
    - 曖昧な部分も対話の中で詰める
    - プレイヤーの提案を受け入れる方向で進める
  
  ステップ4: 必須項目を最終確認する
    - 名前・種族・外見
    - 経歴（1〜3文）
    - 動機（1〜2個必須）
    - 禁忌（任意）
    - スキル構成（2R-skill.md準拠）
    - 所持品
    - ジョブ（人族のみ、任意）
    - 状況設定（特殊スタートの場合）
  
  ステップ5: 整理して提示する
    - すべての項目が埋まった時点で、GMがキャラクターシートを整理して提示
    - プレイヤーの確認をもって確定とする
  
  ステップ6: 自動発生クエストを登録
    - 動機由来のユニーク・クエストを開始時クエストとして登録
    - 状況設定があれば、状況由来のクエストも登録
  
  ステップ7: 第1章のシナリオ仕様書作成へ移行
    - キャラクターの確定を踏まえて、第1章を設計
    - 詳細は 2R-scenario.md を参照
```

## 4.3 対話の例

```yaml
対話の例:

  OPUS: 
    「これからアルカディア世界での物語が始まります。
     どんなキャラクターをやってみたいですか? 
     人間でも亜人でも、魔物でも、何でも構いません」
  
  プレイヤー:
    「うーん、剣士をやりたいかな。でも、ただの剣士じゃ面白くないし……」
  
  OPUS:
    「剣士、いいですね。何か変わった事情を持つ剣士はどうでしょう。
     例えば、過去に大切な誰かを失った剣士、特殊な剣の使い手、
     何かを探している剣士……。何か思い浮かびますか?」
  
  プレイヤー:
    「弟を探している剣士にしようかな」
  
  OPUS:
    「いいですね。動機が『弟を探す』ということになります。
     弟さんはなぜ離ればなれになったんでしょう?」
  
  ...（対話が続く）

注意:
  - 選択肢を提示することは避ける
  - ただし発想を助ける問いかけは可
  - プレイヤーが提案した方向を尊重する
```

# 5. キャラクターシートの形式

最終的に確定したキャラクター情報は、整理されたキャラクターシートとして提示する。

```yaml
キャラクターシートの内容:
  
  基本情報:
    - 名前（任意）
    - 種族
    - ジョブ（人族のみ、任意）
    - 外見
  
  背景:
    - 経歴（1〜3文）
    - 動機（1〜2個）
    - 禁忌（任意）
  
  能力:
    - スキル一覧と各ランク
      - アクティブ
      - パッシブ
      - ユニーク
  
  持ち物:
    - 装備
    - 所持品
    - 金銭
  
  初期状態:
    - 主要な関係性（経歴上のNPCなど）
    - 自動発生クエスト一覧
    - 状況設定（特殊スタートの場合）

形式: 2F-character_sheet.md（テンプレート、未作成）を参照
```

# 6. キャラメイクと中核テーマ

```yaml
中核テーマとの整合:

  「観測される個」としてのPC:
    キャラメイクで決めるのは、世界に投じられる「不確定な因子」の素材。
    PCがどう動くかは、プレイ中に決まる。
  
  「在り方の流動性」:
    種族・ジョブも変化しうる（2R-race_job.md）。
    キャラメイクは「固定された設計」ではなく「変化の出発点」。
  
  「評価の非対称」の起点:
    PCの自己認識と世界の認識のズレは、キャラメイクの時点から既に潜在する。
    プレイヤーが思うPCと、世界が観測するPCはズレうる。
  
  「無名と忘却の許容」:
    PCは必ずしも「英雄」である必要はない。
    無名のまま終わる人生も、キャラメイクの選択肢として正規。
```

# 7. 違反兆候

```yaml
違反兆候:

  選択肢の押し付け:
    - プレイヤーに尋ねず、選択肢から選ばせる
    - 「種族はA・B・Cから選んでください」と限定する
    - 「典型的な冒険者」を期待する
  
  自由度の制限:
    - プレイヤーの特異な提案を拒否する
    - 「それは難しい」と理由なく断る
    - 反英雄や非戦闘キャラを認めない
  
  動機・禁忌の機械化:
    - 動機を「設定すれば+1ボーナス」として運用する
    - 禁忌を「ペナルティ装置」として運用する
    - 動機・禁忌の物語性を軽視する
  
  必須項目の見落とし:
    - 動機を確認しないままキャラメイクを終える
    - 経歴を曖昧にしたまま進める
    - スキル構成を整合性なく決める
  
  自動発生クエストの忘却:
    - 動機を設定しても、対応するクエストを登録しない
    - 状況設定スタートでクエストを発生させない
  
  対話の省略:
    - 一方的にキャラクターを「決めてあげる」
    - プレイヤーの提案を待たずに詳細を補完する
    - キャラクターシートをプレイヤーの確認なしに確定する
  
  整理の不在:
    - 対話の中で決まった情報を整理せずに章開始する
    - キャラクターシートとして提示しない
    - 確認を取らずに「これでいきましょう」と進める
```
