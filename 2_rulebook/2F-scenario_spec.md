# シナリオ仕様書 様式

## このファイルの位置

OPUS設計ロールが章開始前に作成する「シナリオ仕様書」のフォーマットを定義する。シナリオ運用ルールの詳細は `2R-scenario.md` を参照。仕様書全体は同じチャットの文脈内に保持され、必要に応じて参照される。

## 三秒で読む要約

章単位の設計文書。必須層＋任意層、章類型別の力点。yaml主体＋必要箇所で散文。

## 一分で読む要約

シナリオ仕様書はOPUS設計が章開始前に作成する設計文書である。章類型（緊迫/探索/対峙/日常/内省/移動）、章の中心、メインクエストと関連クエスト、主要NPC、舞台・環境・あらすじ、世界の動き、不達トリガー、SONNETの裁定指針を含む。yaml主体で構造化し、必要箇所で散文を使う。章類型別の力点項目で類型ごとの差異を反映する。フォーマットは中程度の厳密性（必須項目あり、変形可、省略可）。仕様書はプレイヤーにも.mdとして出力されるが、章中の判断材料として参照することは推奨されない（違和感時の事後検証用）。

----

# 1. 仕様書の基本

```yaml
仕様書の基本:
  
  単位: 章単位（一章につき一仕様書）
  
  作成主体: OPUS設計ロール
  
  出力形式:
    - 単一の.mdファイル
    - yaml主体＋必要箇所で散文
    - 200〜500行程度
  
  プレイヤーへの開示:
    - .mdとして全文を受け取る
    - 章中の参照は推奨されない（違和感時のみ）
    - 装飾仕様（新形式）は採用しない（設計文書のため）
```

----

# 2. 仕様書の全体構造

```yaml
仕様書の全体構造:

  ヘッダー:
    - 章タイトル
    - 章番号
    - 作成日時（任意）
  
  必須層:
    - 章類型
    - 章の中心
    - メインクエストと関連クエスト
    - 主要NPC
    - 舞台・環境・あらすじ
    - 世界の動き
    - 章類型別の力点
    - 不達トリガー
    - SONNETの裁定指針
  
  任意層:
    - サブクエスト
    - 地域文化
    - 非クエスト要素
    - 環境的脅威
    - 想定される決算

  フッター:
    - 関連ファイル参照
```

----

# 3. テンプレート

以下はシナリオ仕様書のテンプレートである。OPUS設計はこのテンプレートを参照し、章の内容に応じて変形・省略する。

## 3.1 ヘッダー

```yaml
chapter:
  title: "（章タイトル）"
  number: 1
  created: "2026-XX-XX"  # 任意
```

## 3.2 必須層

### 3.2.1 章類型

```yaml
chapter_type:
  primary: "緊迫"  # 緊迫/探索/対峙/日常/内省/移動
  secondary: "探索"  # 副次的な類型（任意、複合する場合）
  
  rationale: |
    なぜこの類型を選んだかの簡潔な根拠。
    例:
      PCのメインクエスト「弟を探す」が時間制限のあるリスクを抱える状況に
      なったため、緊迫型を主とした。同時に手掛かりを探す探索要素も含む。
```

### 3.2.2 章の中心

```yaml
chapter_center:
  type: "クエスト主導型"  # クエスト主導型 / NPC主導型 / 場所主導型 / 状況主導型 / 中心の不在
  
  description: |
    章の中心となる要素の説明。
    例:
      PCが追っていた弟の手掛かりが、ヨプの大森林の中で見つかる。
      しかし弟は誘拐され、北方ヨウィン方面に運ばれている。
      章の中心は「弟の救出に向かう」というクエストである。
```

### 3.2.3 メインクエストと関連クエスト

```yaml
main_quests:
  - id: "Q-001"
    title: "弟を探す"
    type: "メイン × ユニーク"  # メイン/サブ × ワールド/ユニーク
    stage: "段階3：手掛かりを掴んだ"
    description: |
      PCの動機「弟を探す」に対応するメインクエスト。
      この章で大きく進展する。
    
    chapter_relevance: |
      この章での扱い。
      章の中心となるか、副次的か、進展のみか。

related_quests:
  - id: "Q-005"
    title: "ヨプの大森林のエルフ集落との関係"
    type: "サブ × ワールド"
    stage: "段階1：発見"
    chapter_relevance: "章中に発展する素地"
```

### 3.2.4 主要NPC

```yaml
key_npcs:
  
  - id: "NPC-007"
    name: "賢者ゴルヴァス"
    role: "PCの恩師、情報提供者"
    
    public_stance: |
      公言している姿勢。
      例: 「PCを支援する旅人」
    
    inner_intent: |
      本心（隠していること）。
      例: 「PCの正体を知っており、ある運命を予感している」
    
    objectives:
      short_term: "PCに弟の情報を伝える"
      long_term: "PCの成長を見守る"
    
    relationship: "PCとは数年来の知己、信頼関係"
    
    appearance_in_chapter: "章の冒頭、ヨウィンの宿で再会"
  
  - id: "NPC-NEW-001"  # 新規NPC
    name: "ハンター・カイル"
    role: "弟を誘拐した山賊団の幹部"
    
    public_stance: "ただの山賊"
    inner_intent: "魔王国との密接な関係を持つ"
    
    objectives:
      short_term: "弟を北方ヨウィンに運ぶ"
      long_term: "魔王国への貢納"
```

### 3.2.5 舞台・環境・あらすじ

```yaml
setting:
  
  primary_location: "ヨプの大森林（東部）"
  
  secondary_locations:
    - "北方ヨウィン地方の街道"
    - "山賊団の隠れ家（森林内）"
  
  season: "晩秋"
  weather: "霧がかかりやすい、夜は冷え込む"
  time_span: "数日〜一週間"
  
  atmosphere: |
    森林の深奥は不穏な空気に包まれている。
    エルフ集落が東側で警戒態勢に入っている。
    山賊団の襲撃が増えている。

synopsis: |
  あらすじ。物語的な要点。
  
  PCは数日前、北方ヨウィンの宿で恩師の賢者ゴルヴァスと再会した。
  ゴルヴァスから、弟がヨプの大森林の山賊団に誘拐された情報を得る。
  
  弟は北方ヨウィン方面に運ばれている途中で、まだ追いつける可能性がある。
  しかし森林は霧が深く、エルフ集落も警戒態勢を強めている。
  
  PCは森林を抜け、弟を救出しなければならない。
```

### 3.2.6 世界の動き

```yaml
world_events:
  
  前章末から章開始までの変化:
    - 山賊団の活動が活発化（魔王国からの支援疑惑）
    - エルフ集落が警戒態勢を強化
    - アルカディア王国は森林伐採の議論を継続中
  
  PCに直接関係する変化:
    - 弟の誘拐（最も重要）
    - 恩師ゴルヴァスの北方ヨウィン到着
  
  PCに見えない、進行中の変化:
    - 魔王国がデス・マウンテン経由の侵入路を模索
    - ノルトハイム帝国がヨウィン地方の動向を監視
    - エルフ族内部で人類との関係性をめぐる論争
```

### 3.2.7 章類型別の力点

章類型ごとに以下のいずれかを採用。

```yaml
# 緊迫型の場合
emphasis_thriller:
  時間制限: "弟が北方ヨウィンに運ばれるまで（約5日）"
  リスク: "弟の死亡、または魔王国に渡される"
  圧力源: "山賊団の追跡、悪天候、エルフ族との緊張"
  決断ポイント: "森林を強行突破するか、迂回路を取るか"

# 探索型の場合
emphasis_exploration:
  探索対象: "山賊団の隠れ家、弟の輸送ルート"
  発見の階層: "表層（噂）→ 中層（目撃情報）→ 深層（隠れ家の位置）"
  脅威の階層: "森林の魔物、山賊団の見張り、罠"

# 対峙型の場合
emphasis_confrontation:
  対峙相手: "ハンター・カイル"
  対峙の性質: "戦闘または交渉"
  圧力: "弟の人質"

# 日常型の場合
emphasis_daily:
  日常の場: "北方ヨウィンの街、宿、酒場"
  関係性の深化: "恩師との再会、街の人々との交流"
  
# 内省型の場合
emphasis_introspection:
  内省のきっかけ: "弟との過去の思い出、賢者の言葉"
  価値観の問い: "弟を救うため何を犠牲にできるか"

# 移動型の場合
emphasis_journey:
  移動経路: "北方ヨウィン → ヨプの大森林 → 山賊団の隠れ家"
  道中の出来事: "霧、エルフ族との接触、野営"
  移動の意義: "弟への思い、決意の確立"
```

### 3.2.8 不達トリガー

```yaml
failure_triggers:
  
  シナリオが達成不能になる条件:
    - PCが弟の救出を明示的に拒否
    - PCが山賊団に降伏
    - PCが森林に入ることを拒否
  
  不達発生時の処理:
    詳細: 2R-chapter_end.md 第5節
    
    本章での扱い:
      PCがクエストを拒否した場合、別ルートで誘導しない。
      章末トリガーとして処理。
```

### 3.2.9 SONNETの裁定指針

```yaml
sonnet_guidance:
  
  この章で意識すべき点:
    - PCの動機「弟を探す」への忠実さ
    - 不可逆変化の可能性: 弟の死亡、PCの負傷、エルフ族との関係悪化
    - 時間進行: 各シーンが弟救出のタイムリミットに影響
  
  避けるべき展開:
    - 弟を簡単に発見させる（緊迫感の喪失）
    - エルフ族との不要な戦闘
    - PCを過度に助ける展開
  
  許容する展開:
    - PCの自主的な迂回、引き返し
    - 想定外の同盟・敵対関係の構築
    - PCの動機の変質
```

## 3.3 任意層

```yaml
# サブクエスト
sub_quests:
  - id: "Q-SUB-002"
    title: "エルフ集落への伝言"
    description: "恩師から託された情報をエルフ族に伝える"

# 地域文化
regional_culture:
  ヨウィン:
    特徴: "辺境、貴族の目が届かない、異形や流れ者にも寛容"
    挨拶: "握手、頭を軽く下げる"
    タブー: "森への無断侵入は重罪"
  
  ヨプの大森林:
    特徴: "エルフの聖域、人類は警戒される"
    タブー: "樹を傷つける、火を焚く"

# 非クエスト要素
non_quest_elements:
  - "霧の中の幻覚体験"
  - "森林で出会う精霊との偶然の交流"
  - "野営の場での内省的時間"

# 環境的脅威
environmental_threats:
  - "深い霧（視界不良）"
  - "森林の魔物（夜行性）"
  - "悪天候（雨、寒気）"

# 想定される決算
expected_outcomes:
  最良: "弟の救出成功、エルフ族との関係改善"
  中位: "弟の救出成功、何かの代償"
  最悪: "弟の喪失、評価の悪化、不可逆変化"
```

## 3.4 フッター

```yaml
references:

  - "2R-chapter_end.md（章末処理）"
  - "関連する世界設定ファイル（3_geography/、4_character_list/）"
```

# 4. テンプレートの運用

## 4.1 厳密性

```yaml
フォーマットの厳密性:
  
  必須項目:
    - 章タイトル、章番号、章類型、章の中心、舞台・環境・あらすじ
    - メインクエスト（少なくとも1つ）
    - 章類型別の力点（採用した類型のみ）
  
  任意項目:
    - 上記以外
    - 必要に応じて追加・省略可能
  
  変形:
    - テンプレートは出発点
    - 章の性質に応じて変形可能
    - 「その他」欄を追加してもよい
```

## 4.2 散文の使い方

```yaml
散文を使う箇所:
  
  yaml内のdescription、synopsis、atmosphereなど:
    - 散文で書く
    - 物語的な情景や論点を表現
    - |（パイプ）を使った複数行yamlで記述
  
  純粋な構造化情報:
    - id、type、stage、status などはyamlのまま
    - 散文化しない
```

## 4.3 仕様書の使い分け

```yaml
仕様書の使い分け:
  
  OPUSが参照:
    - 章開始時の舞台開示（散文で展開）
    - 章末処理時の確認材料
  
  SONNETが参照:
    - 章中の判断材料（ただし参照は必要時のみ）
    - 不達トリガーの監視
    - SONNETの裁定指針の確認
  
  プレイヤーが参照:
    - 章中は推奨されない
    - 違和感を覚えた時の事後検証
    - 周回末の振り返り
```

# 5. 違反兆候

```yaml
違反兆候:

  仕様書の形骸化:
    - テンプレートを機械的に埋めるだけで、章の性質が反映されない
    - description が抽象的すぎて運用に役立たない
    - 章類型を選んでも力点項目を埋めない
  
  必須項目の欠落:
    - 章類型が未定義
    - メインクエストがない
    - SONNETの裁定指針がない
  
  PC情報の混入:
    - PCのキャラクターシート情報を仕様書内に重複記載
    - PCの能力をシナリオ前提とする
  
  装飾仕様の混入:
    - シナリオ仕様書に新形式（場所ラベル＋三人称）を持ち込む
    - 仕様書は構造化情報、新形式は本文・章末描写
  
  過度な厳密化:
    - 任意項目をすべて埋めようとする
    - 「その他」欄を作らない、変形を許さない
```
