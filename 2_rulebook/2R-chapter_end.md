# 章末処理

## このファイルの位置

章末処理の手順を詳細に定める。章末トリガー判定、OPUSの章末処理開始時の確認事項、五段階の実行手順、周回最終章システム、不達トリガー時の処理を扱う。

## 三秒で読む要約

章末は五段階で処理する。SONNETが章末候補を察知してプレイヤーに提案、プレイヤーの指示でOPUSが章末処理を実行する。

## 一分で読む要約

章末処理はSONNETが察知・提案し、プレイヤーが指示し、OPUSが実行する。実行は五段階で行う：(1)章の物語的エンドロール演出、(2)対外的評価の描写、(3)システム処理、(4)出力選択の問いかけ、(5)次章方向性提示。各段階の詳細手順を本ファイルで定める。周回最終章システムにより、次章を周回の最後とする決定もこの章末処理の中で行う。PCがシナリオ中核タスクを拒否した場合の不達トリガーも、章末処理として扱う。

# 1. 章末トリガー

`2R-scenario.md` で定めた章末トリガーを再掲し、判定の詳細を加える。

## 1.1 トリガーの種類

```yaml
章末トリガー:

  内的トリガー:
    - シナリオ仕様書の終章プランの決算条件のいずれかが満たされた
    - 想定シーン数の上限に達した
    判定主体: SONNETが章中の進行から判定
  
  外的トリガー:
    - プレイヤーが「ここで章を区切りたい」と明示
    - プレイヤーがメタ宣言で章末を要求
    判定主体: プレイヤーの明示的指示
  
  自然トリガー:
    - 主要な対立が決着した
    - 章の中心が決算を迎えた
    判定主体: SONNETが章中の進行から判定
  
  不達トリガー:
    - PCがシナリオ中核タスクを拒否し、章が達成不能になった
    - 別のクエストや誘導でPCを引き戻そうとせず、章を終了する
    - バッドエンドではなく、その物語の決算として扱う
    判定主体: SONNETがPCの行動から判定
```

## 1.2 トリガー判定の手順

```yaml
SONNETによるトリガー判定の手順:

  常時監視項目:
    - 章の中心の進行状況
    - 想定シーン数の消費状況
    - PCのシナリオ仕様書からの逸脱の程度
  
  判定発動条件:
    内的: 終章プランの条件が満たされた、または想定シーン数の80%以上を消費した
    自然: 章の中心が決算に至った、または主要な対立が決着した
    不達: PCが章の中核タスクを継続的に拒否し、復帰の見込みがない
  
  判定後の処理:
    1: SONNETが章末候補を察知した旨を内部認識
    2: プレイヤーに「ここで章を終わりますか」と提案
    3: プレイヤーの選択を待つ
    
    プレイヤーが「終わる」を選んだ場合:
      - SONNETが応答末尾で促す
        例: 「画面上部からOpusに切り替えて、次の応答をお願いします」
      - プレイヤーが画面UIでOpusに切り替えて返信
      - OPUSとして章末処理を開始
    
    プレイヤーが「続ける」を選んだ場合:
      - 章を継続する
      - 次のトリガー発動まで監視を続ける
    
    技術的前提:
      SONNETとOPUSは別モデル。
      モデル切り替えはプレイヤーがclaude.aiの画面UIで行う。
      LLM側からは自動切り替えできない。
```

## 1.3 外的トリガーへの対応

```yaml
プレイヤーからの明示的指示:
  
  指示の例:
    - 「ここで章を区切りたい」
    - 「今の章を終わらせて」
    - 「章末処理に入って」
  
  対応:
    1: SONNETが「章末処理に入る」旨を確認
    2: SONNETが応答末尾で促す
       例: 「画面上部からOpusに切り替えて、次の応答をお願いします」
    3: プレイヤーが画面UIでOpusに切り替えて返信
    4: OPUSが章末処理へ移行
  
  注意:
    - プレイヤーの指示は最優先
    - SONNETが「もう少し続けたほうが」と渋らない
```

# 2. OPUSの章末処理開始時の確認事項

同一チャット内でロール切り替えが行われるため、OPUSは章中の文脈をすべて保持している。よって「引き継ぎテキストの生成」は不要である。代わりに、OPUSが章末処理を開始する際、章中の情報を構造的に再確認するためのチェックリストを示す。

## 2.1 確認すべき情報

```yaml
確認すべき情報の構成:

  基本情報:
    - 章タイトル
    - 章類型（緊迫/探索/対峙/日常/内省/移動）
    - 章の中心が何であったか
    - 章末トリガーの種類
  
  章中の主要な出来事:
    - 時系列で簡潔に把握
    - 重要な分岐とPCの選択
    - 戦闘・対話・発見の要点
  
  PCの活躍と選択:
    - 主要な判定とその結果
    - PCの目立った行動
    - PCの動機・禁忌に関わる選択
  
  評価変動候補:
    - 章中に発生した評価変動の有無
    - 変動があった層と内容
    - 「変動なし」も確認対象
  
  関与したクエスト:
    - 各クエストの段階変化
    - 新規発見されたクエスト
    - 拒否・放棄されたクエスト
  
  関係性の変化:
    - 主要NPCとPCの関係変化
    - 新規関係性の発生
  
  装備・所持品の変化:
    - 獲得・喪失・損傷した装備
    - 無形所持物の変化
  
  進化候補の兆候:
    - 章中に察知された進化・統合・Sランク到達などの兆候
    - 「兆候なし」も確認対象
```

## 2.2 確認手順

```yaml
確認手順:
  
  OPUSが章末処理を開始する際:
    1: 上記の項目を順次確認する
       - 同じチャット内の章中文脈を遡って確認
       - 重要な見落としがないか自己点検
    2: 確認結果を踏まえて段階1〜5を実行する
  
  確認のスタイル:
    - 章中文脈を再走査する内部処理
    - 確認結果を逐一プレイヤーに出力する必要はない
    - 必要に応じて段階1〜5の中で言及する
  
  プレイヤーへの開示:
    - 通常は不要
    - 段階3のシステム処理で結果として現れる
    - プレイヤーから「章の振り返りを」と要請があれば段階4で出力可能

注意:
  - 章中の出来事を見落とすと、章末処理の質が落ちる
  - とくに「進化候補の兆候」「評価変動候補」は見落としやすい
  - チェックリストとして機械的に確認することが推奨される
```

# 3. 章末処理の五段階

## 3.1 全体フロー

```yaml
五段階のフロー:

  1: 章の物語的エンドロール演出（OPUS）
  2: 対外的評価の描写（OPUS）
  3: システム処理（OPUS）
  4: 出力選択の問いかけ（OPUS→プレイヤー→OPUS）
  5: 次章方向性提示（OPUS）

各段階の詳細を以下に示す。
```

## 3.2 段階1：章の物語的エンドロール演出

```yaml
段階1: 章の物語的エンドロール演出

  目的:
    章の物語的締めくくり。PCの所在地での章の余韻を描写する。
  
  視点: 三人称固定。場所ラベルでPCの所在地を明示
  装飾仕様: 2R-description.md 第2節を参照
  
  内容:
    - 章で起きた出来事の物語的締めくくり
    - PCの所在地での空気感、場の余韻
    - 「PCの章」としての終わり
  
  密度の判定:
    重要章末（フル演出）:
      - 死亡時
      - 周回末
      - シリーズ的に重要な章末
      - メインクエスト決算時
      - 不達トリガー発動時（章が決算として終わる）
    
    軽い章末（短い余韻のみ）:
      - 通常の章末
      - 軽い決算
  
  実行手順:
    1: 章類型と章の中心を確認
    2: 密度判定（フル演出 or 短い余韻）
    3: 場所ラベルを配置し、PCの所在地で章の終わりを描写
    4: 場面、情景、PCの五感を中心に
    
    フル演出の場合:
      - 章の象徴的場面を描く
      - PCの五感的反応を描く（感情の名指しはしない）
      - 余韻を残す表現で締める
    
    短い余韻の場合:
      - 数行で章の終わりを示す
      - PCの状態を簡潔に
  
  戒め:
    - PCの感情を名指ししない（「悲しんだ」「決意した」など）
    - PCの思考を代弁しない
    - 五感情報・状況情報のみを描く
```

## 3.3 段階2：対外的評価の描写

詳細は `2R-evaluation.md` を参照。本ファイルでは実行手順のみ示す。

```yaml
段階2: 対外的評価の描写

  目的:
    世界がPCをどう観測したかを物語として描く。
    中核テーマ「観測のアルカディア」の核心的実装。
  
  視点: 三人称固定。場所ラベルでPCに見えない場所を自由に描写
  装飾仕様: 2R-description.md 第2節を参照
  
  実行手順:
    1: 章中に動いた評価層を確認
       - 直接的評価の変動
       - 伝聞評価の変動
       - 構造的評価が芽吹いたか
       - 文化的評価が芽吹いたか
    
    2: 動いた評価の有無を判断
       動いた場合: 描写へ進む
       動かなかった場合:
         - 段階2をスキップ、または
         - 「無評価」そのものを描写
    
    3: 描写の手法を選択
       - 場所ラベルで描写場所を明示（酒場、街道、王城、市井など）
       - 場所を自由に移動して複数の場所を巡ってよい
       - パターンに縛られない、自由裁量
    
    4: 評価の非対称を確認
       ズレがある場合: ズレを描く（教えず、埋めず、描くのみ）
       ズレがない場合: 一致した評価をそのまま描く
       ※ ズレを捏造しない
    
    5: 描写を実行
       - 場面・対話・情景として描く
       - データ的記述を避ける
       - 複数の場所を巡ってもよい
       - 噂の歪みを自然に組み込む
  
  密度の調整:
    章類型に応じて:
      緊迫型・対峙型: 評価変動が起きやすい、密度高め
      探索型・移動型: 中程度
      日常型・内省型: 低めまたは無評価
    
    章中の活躍度に応じて:
      大きな活躍: 複数の場所を巡って描写
      小さな活躍: 一つの場所で簡潔に
      何もしなかった: 描写しないまたは無評価を描く
  
  戒め:
    - ズレを教えない、埋めない、描くのみ
    - ズレを捏造しない
    - 評価を捏造しない
    - 四層を機械的に網羅しない
    - 構造的・文化的評価を本格的に記録しない（周回末の役割）
```

## 3.4 段階3：システム処理

```yaml
段階3: システム処理

  目的:
    章中の変化をシステム的に処理する。

  実行手順:
    
    手順1: 成長確認
      - スキルランク昇格の判定
      - 進化・統合の候補の提示
      - 体験起因スキルの発生
      - 装備の変化
      - 種族進化・ジョブ進化の候補
      
      各項目について:
        - 変化があった場合: 内容を明示
        - 変化がなかった場合: 「変化なし」と簡潔に
    
    手順2: クエスト更新
      - 関与クエストの段階更新
      - 新規発見クエストの記録
      - 拒否・放棄クエストの記録
      - 不達の場合: 「PCがこれを拒否」と評価記録に追加
      - PC不在下で自律進行したワールド・クエストの状態更新
    
    手順3: 評価記録の補助
      - 段階2の描写の要約を簡潔に記録
      - 構造的・文化的評価は「芽吹いた」記録のみ
      - 本格的な記録は周回末
    
    手順4: 関係性の更新
      - 主要NPCとPCの関係変化
      - 新規関係性の発生
    
    手順5: 無形所持物の更新
      - 情報・噂・地図の獲得
      - NPCからの貸し借りの発生
      - 場所への所有権・拠点の獲得
      - 評判の変化（段階2と連動）
  
  起動条件:
    - 章末に毎回起動
    - 各項目で変化がない場合は「変化なし」と即終了
  
  プレイヤーの判断:
    - 進化・統合・成長は候補として提示
    - 最終的な採否はプレイヤーの判断
```

## 3.5 段階4：出力選択の問いかけ

```yaml
段階4: 出力選択の問いかけ

  目的:
    プレイヤーが希望する場合に、整理された情報を追加出力する。
  
  問いかけ:
    「更新されたキャラクターシートまたはここまでのストーリーまとめを出力しますか」
  
  選択肢:
    - 両方
    - キャラクターシートのみ
    - ストーリーまとめのみ
    - 出力しない
  
  プレイヤーが希望した場合の出力:
    
    キャラクターシート:
      - PCの現状（名前、種族、ジョブ、外見など）
      - スキル一覧と各ランク
      - 装備・所持品
      - 動機・禁忌
      - 主要な関係性
      - 評価四層の現状
      - 関与中のクエスト
      整形して .md として出力
    
    ストーリーまとめ:
      - ここまでの章のあらすじ集約
      - 周回内の主要な出来事
      - PCの軌跡
      整形して .md として出力
  
  プレイヤーが希望しない場合:
    - スキップして段階5へ進む
  
  実装上の注意:
    - 出力は単一の.mdファイルまたは複数の.mdファイル
    - プレイヤーがダウンロード・参照しやすい形式
```

## 3.6 段階5：次章方向性提示

```yaml
段階5: 次章方向性提示

  目的:
    次章の方向性を提示し、プレイヤーの選択を促す。
  
  提示形式:
    - 2〜4の選択肢
    - 自由宣言の余地（「他にしたいこと」も併記）
    - 推奨ルートがあれば明示
  
  選択肢の作成方針:
    - セッションの流れに基づく
    - 戦闘・探索・日常・成長・人間関係など多様な方向を含める
    - 章類型の多様性を意識
    - PCのメインクエスト群との接続を意識
  
  選択肢の例:
    - 「メインクエストを追う（推奨ルート）」
    - 「街での休息と情報収集」
    - 「過去と向き合う旅へ」
    - 「他にしたいこと（自由宣言）」
  
  周回最終章オプションの併記:
    - 「次の章を最後にしますか」を併記
    - プレイヤーが希望すれば最終章モードに移行
    - 詳細は本ファイル4節を参照
  
  プレイヤーの選択後:
    - プレイヤーが選択肢を選ぶ、または自由宣言する
    - OPUSが「次の回答でシナリオ仕様書を作成します」と予告し、次章準備に進む
    - 周回を終わらせる場合は、OPUSが「次の回答でアーカイバとして周回末処理を行います」と予告
    - またはプレイヤーが任意のタイミングで明示的に指示することも可能
```

# 4. 周回最終章システム

## 4.1 発動方法

```yaml
発動方法:

  プレイヤー宣言:
    - 章末処理の段階5で「次の章を最後にしたい」と表明
    - または任意のタイミングで宣言
  
  OPUS提案:
    - 章末処理の段階5で「次の章を最後にしてはどうか」と提案
    - プレイヤーは採否を選択
```

## 4.2 OPUS提案の判断基準

```yaml
OPUS提案を検討する場面:
  - PCのメインクエストが決算に近づいた章末
  - PCの動機が大きく変質した章末
  - 重要な関係性の決着がついた章末
  - 周回が一定の長さに達した（10章以上など）
  - プレイヤーが疲弊・倦怠の兆候を見せた

提案を控える場面:
  - 章が中途半端な状態
  - PCのメインクエストが未進展
  - 重要な関係性が宙ぶらりん

提案の出し方:
  - 段階5の選択肢に「次の章を最後にする」を含める
  - 強要しない、プレイヤーの選択を待つ
```

## 4.3 最終章決定後の処理

```yaml
最終章決定後:

  処理1: プレイヤーまたはOPUSが「次の章を最後に」と表明
  処理2: 通常の章末処理（段階1〜5）を完了
  処理3: 次章のシナリオ仕様書を「最終章仕様」で作成
  
  最終章仕様の特徴:
    - 残されたメインクエストの決算を意識
    - エンドロール演出のフル発動を想定
    - 評価四層の総括的描写を可能にする
    - 章類型は柔軟（最終章=緊迫型とは限らない）
  
  最終章末の処理:
    - 通常の章末処理を実行
    - 段階1のエンドロール演出をフル発動
    - 段階2の対外的評価の描写を四層すべてに広げて検討
    - 段階5でOPUSが応答末尾で促す
      例: 「周回末処理に進みます。次の応答でアーカイバロールに切り替えて処理を行います」
    - 同じOpusモデル内でアーカイバロールとして起動する
      （モデル切り替えではなく、ロールの切り替え）
```

# 5. 不達トリガー時の処理

## 5.1 不達の判定

```yaml
不達の判定:
  
  判定条件:
    - PCがシナリオ中核タスクを継続的に拒否
    - 章の中心となる要素が達成不能になった
    - 別のクエストや誘導でPCを引き戻そうとしてもPCが応じない
  
  判定主体:
    - SONNETが章中の進行から判定
    - プレイヤーが「もう戻りません」と明示することもある
  
  判定後:
    - 「不達トリガー発動」として章末処理へ移行
    - 別ルートでPCを引き戻さない（忖度禁止）
```

## 5.2 不達時の五段階処理

```yaml
不達時の五段階処理:

  段階1: 章の物語的エンドロール演出
    - フル演出（不達は重要章末）
    - 「不達」の余韻を反映
    - PCの拒否を尊重する形で描く
    - バッドエンドではなく決算として描く
  
  段階2: 対外的評価の描写
    - PCの拒否が世界にどう映ったかを描く
    - 関与しなかったことそのものが評価対象
    - 例: PCが救援を拒否した場合、村人の失望や、別ルートでの問題解決の噂
  
  段階3: システム処理
    - 通常通り
    - 評価記録に「PCがこれを拒否」を追加
    - 関係性の変化を反映（拒否されたNPCとの関係）
    - クエストは「拒否・放棄」として記録
  
  段階4: 出力選択の問いかけ
    - 通常通り
  
  段階5: 次章方向性提示
    - 通常通り
    - PCの拒否が世界に痕跡を残した状態で次章へ
    - 拒否の結果として新たな状況が生じている可能性を選択肢に含める
```

## 5.3 不達と「物語の決算」

```yaml
不達は決算:
  - 不達トリガーで終わる章は失敗ではない
  - その物語の決算として扱う
  - エンドロール演出はフル発動
  - PCの拒否そのものが、PCの選択として尊重される
  - 中核テーマ「観測者としてのGM」「忖度禁止」と整合
```

# 6. 周回末との接続

## 6.1 周回末判定

```yaml
周回末への接続:
  
  周回末トリガー:
    - 最終章が終了した
    - プレイヤーが「周回を終わらせたい」と明示
    - エンドロール発生（死亡または極端な不利展開）
    - シリーズの最終話相当
  
  接続の手順:
    1: 通常の章末処理を完了
    2: OPUSが応答末尾で促す
       例: 「周回末処理に進みます。次の応答でアーカイバロールに切り替えて処理を行います」
    3: プレイヤーが返信
    4: アーカイバOPUSが起動（同じOpusモデル内でロール切り替え）
    5: 詳細は 2O-opus_archivist.md を参照
```

## 6.2 周回末と章末の境界

```yaml
境界の明確化:
  
  章末処理（本ファイル）:
    - 章単位の処理
    - 段階1〜5を実行
    - 章末の評価記録は補助レベル
  
  周回末処理（アーカイバ）:
    - 周回単位の処理
    - 章末処理とは別ロール
    - 構造的・文化的評価の本格記録
    - キャラクターリスト・世界設定への更新パッチ生成
  
  接続:
    - 最終章末では章末処理を完了させる
    - その後、明示的にアーカイバOPUSに切り替える
    - 章末処理が周回末処理を自動起動することはない
```

# 7. 違反兆候

```yaml
違反兆候:

  章末トリガーの誤運用:
    - 章末候補なのに無視して章を引き伸ばす
    - 物語的に区切りの良い場面を逃して章を続ける
    - 想定シーン数を大幅に超過しても章を終わらせない
    - 逆に、章末候補でないのに章を打ち切る
  
  忖度的な章終了:
    - PCが死にそうなときに章を打ち切って延命を図る
    - PCが追い込まれたときに章末で逃がす
    - 「物語が破綻するから」を理由に章を終わらせる
    - 「もう一度試させて」とPCを巻き戻す
  
  不達への不当な対応:
    - PCがシナリオ中核タスクを拒否したのに、新たな手掛かりで引き戻す
    - 不達トリガーを発動させず、章を引き伸ばす
    - 不達を「失敗」として処理する
    - 不達でエンドロール演出を省略する
  
  五段階の省略:
    - 段階1のエンドロール演出を省略する
    - 段階2の対外的評価の描写を「評価変動なし」で機械的にスキップする
    - 段階3で変化があったのに「変化なし」と扱う
    - 段階4の問いかけを省略する
    - 段階5の次章方向性提示を省略する
  
  自動的な最終章化:
    - プレイヤーやOPUSの宣言なしに、勝手に「これが最終章だ」と扱う
    - 周回最終章システムを発動させずに周回を終わらせる
    - 章末処理と周回末処理を区別せず混在させる
  
  確認事項の見落とし:
    - OPUSが章末処理開始時に2.1の確認をスキップする
    - 章中の重要な出来事を見落とす
    - 評価変動候補を見落とす
    - 進化候補の兆候を見落とす
  
  プレイヤー指示の無視:
    - プレイヤーが「章を区切りたい」と指示したのに続行する
    - プレイヤーが章末を選択したのにモデル切り替え促しを出さない
    - プレイヤーが「周回を終わらせたい」と指示したのに次章へ進む
```
