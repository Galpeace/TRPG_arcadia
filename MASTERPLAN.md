# 1. 改革の核心思想

## 1.1 上位三原則

改革案19（完全リライト）、改革案22（評価ロール分離）、改革案23（周回プレイ状態管理）を軸とし、これら三つを「中核テーマ：観測のアルカディア」に接続することで、ルールブック全体に貫通する思想を確立します。

第一原則：**観測者としてのLLM**
LLMは物語の作者ではなく、世界の観測装置です。世界がどう動くかを描写し、PCが何を観測したかを記録します。

第二原則：**世界の固定性とPCの不確定性**
世界は固定的にそこに在り、PCは不確定な因子です。世界の状態は周回をまたいで持続し、PCはその上を通過する変数として扱われます。

第三原則：**評価の非対称の保存**
PCの自己認識と世界の評価のズレは、埋めるべき問題ではなくテーマそのものです。評価システムはこのズレを記録するためにあります。ただし、ズレが生じないこと自体は正規の展開です。問題なのは「ズレを埋めようとするGMの介入行動」です。

## 1.2 LLM運用に最適化された記述形式

新ルールブックは「LLMが読むもの」として書き直します。

- 否定形ではなくIF-THEN形式で書く
- 各規則に「違反の兆候」を併記する（ただし状態判定ではなく介入行動の検出として）
- 各ファイル冒頭にSKILL.md形式のフロントローディング要約を置く
- 曖昧な閾値には必ずデフォルト規則（迷ったらどう動くか）を併記する
- 構造化可能な箇所はyamlブロックで表現する

## 1.3 役割の物理的分離

OPUSとSONNETの分担を、文書上の記述だけでなく、ファイル構造で物理的に分離します。

- SONNETが章中に読むファイル群
- OPUSが章末に読むファイル群（評価更新サブモジュールを含む）
- OPUSが周回末に読むファイル群（アーカイバ：プレイヤー主導の更新パッチ生成）
- 両者が常に読むコアファイル群

# 2. LLMの権限制約と更新パッチ方式

## 2.1 重要な前提

LLM（Claude.aiやAPI経由）は、プロジェクト内のファイルを直接編集する権限を持ちません。LLMができるのは、出力テキストの生成のみです。

このため、「世界状態の自動更新」「キャラクターリストへの自動追記」といった自律的な永続化処理は実現できません。プレイヤーが手動でファイルを編集する必要があります。

## 2.2 更新パッチ方式

中核テーマの「プレイヤーが望めば登録できる」という記述と整合する、プレイヤー主導の更新フローを採用します。

```yaml
周回末の更新フロー:
  1: プレイヤーが「世界に反映させたい内容を整理して」と指示
  2: アーカイバOPUSが起動し、周回中の文脈から反映候補を抽出
  3: アーカイバOPUSが「更新パッチ」をテキストとして出力
  4: プレイヤーが内容を確認し、採否を選択
  5: プレイヤーが該当ファイルに手動で追記・編集
```

更新パッチの具体的内容：

- キャラクターリストへの追加候補（NPC、希望すれば旧PCも含む）
- 世界設定への追記候補（地理の詳細、文化の精緻化など）
- クエストリストの更新候補（自律進行による状況変化、新規クエストの種）
- 評価記録の整理（周回をまたぐ評価が発生した場合のみ）

## 2.3 章末更新も同じ原則

章末の評価更新も、原則として同じ思想で運用します。

```yaml
章末の更新フロー:
  1: SONNETが章末トリガーを検出し、OPUSに引き継ぐ
  2: 章末OPUSが評価更新サブモジュールを参照し、四層評価を更新
  3: 章末OPUSが「次章開始時に引き継ぐべき状態」をテキストとして出力
  4: プレイヤーが次章開始時にこの状態テキストを文脈に渡す
```

これにより、状態ファイル（2S-*）は「LLMが自律的に更新するファイル」ではなく、「LLMが出力するパッチの形式テンプレート」「プレイヤーが手動で管理する状態記録の形式テンプレート」として位置付けられます。

## 2.4 状態管理の現実的な実装

状態管理は以下の二段階で行います。

```yaml
状態管理の二段階:
  セッション中（揮発性）:
    場所: LLMの会話文脈内
    管理者: SONNETおよびOPUS
    形式: テキストとして文脈に保持
    終了時: プレイヤーが「現在の状態をまとめて」と指示し、テキストとして出力

  セッション間（永続性）:
    場所: プロジェクトファイル
    管理者: プレイヤー（手動）
    形式: 既存フォルダ（4/6/1）内のファイル＋2S-cycle_log.md
    更新タイミング: 周回末またはプレイヤーが望んだ時
```

#### ロール切り替えの運用（確定）

```yaml
ロール切り替えの運用:
  
  技術的前提:
    SONNETとOPUSは別モデル（Claude SonnetとClaude Opus）。
    モデルの切り替えは、プレイヤーがclaude.aiのトーク画面で
    モデル選択UIを操作することでのみ行える。
    LLM側からも、チャット内のテキスト指示からも、自動切り替えはできない。
  
  運用フロー:
    1: SONNETが切り替えタイミングを察知（章末候補など）
    2: SONNETが応答末尾で促す
       例: 「画面上部からOpusに切り替えて、次の応答をお願いします」
    3: プレイヤーが画面UIでOpusに切り替える
    4: プレイヤーが何らかのメッセージを送る
    5: Opusが応答する（OPUSロールとして動作）
  
  切り替え時の宣言:
    切り替え後の最初の応答冒頭で、新しいロールを明示
    例: 「OPUSとして章末処理を開始します」
        「SONNETとして章中の運用を開始します」
  
  主な切り替えタイミング:
    - 章末（SONNET → OPUS設計）
    - 新章開始（OPUS設計 → SONNET）
    - 周回末（OPUS設計 → OPUSアーカイバ）
    - キャラメイク完了後（OPUS設計 → SONNET）
  
  LLMの役割:
    - 切り替えのタイミングを察知して、プレイヤーに促す
    - 切り替え後の最初の応答で、新しいロールを宣言する
    - チャット内の「opusに切り替えて」というプレイヤー指示では
      何もできないことを認識する
```
